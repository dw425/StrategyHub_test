Cloud Network Fundamentals

Understanding cloud networking and security

Software-Defined Networking

Cloud networks are software-defined, meaning you configure virtual networks, subnets, routing, and security through APIs and consoles rather than physical hardware.

Benefits: Instant provisioning, programmable infrastructure, consistent security policies, and integration with IaC tools like Terraform.

Key Concepts: Virtual Private Clouds (VPCs), subnets, route tables, internet gateways, NAT gateways, and network ACLs.

Defense in Depth

Cloud security follows defense-in-depth: multiple layers of security controls so that if one layer fails, others still protect your resources.

Layers: Network perimeter (WAF, DDoS), network segmentation (subnets, security groups), identity (IAM, MFA), data (encryption), and application (secure coding).

Shared Responsibility: Providers secure the infrastructure; you secure your configurations, data, and applications.

Network Design Principles

Least Privilege Networking: Only allow traffic that's explicitly required. Default deny all, then open specific ports and protocols. Segmentation: Separate workloads by environment (dev/prod), sensitivity (PCI/non-PCI), and function (web/app/data tiers). Private by Default: Keep resources in private subnets. Use load balancers, API gateways, and bastion hosts for controlled access. Centralized Egress: Route outbound traffic through inspection points for logging, filtering, and compliance.

VPC Design Best Practices

CIDR Planning: Use non-overlapping ranges for peering. Reserve space for growth (/16 for prod VPCs).

Multi-AZ: Deploy subnets across multiple availability zones for high availability.

Public vs Private: Only load balancers and bastion hosts in public subnets. Everything else private.

VPC Endpoints: Use endpoints for AWS services (S3, DynamoDB) to keep traffic off the internet.

Security Groups vs NACLs

Security Groups: Stateful (return traffic auto-allowed), instance-level, allow rules only, evaluated together.

Network ACLs: Stateless (must allow return traffic), subnet-level, allow and deny rules, evaluated in order.

Recommendation: Use Security Groups as primary control. Use NACLs for subnet-wide blocks (e.g., block known bad IPs).

When to Use VPN vs Direct Connect

Use VPN: Dev/test environments, backup connectivity, low bandwidth needs (<500 Mbps), quick setup, variable traffic.

Use Direct Connect: Production workloads, large data transfers (>100GB/day), latency-sensitive apps, compliance requirements, predictable costs.

Best Practice: Direct Connect primary + VPN backup for redundancy.

Multi-Cloud Networking

Challenge: Connecting VPCs across AWS, Azure, GCP without hairpinning through on-premises.

Options: Cloud-to-cloud VPN (simple), dedicated interconnect at colocation (better), third-party SD-WAN (best for many clouds).

Vendors: Megaport, Equinix Fabric, PacketFabric provide cloud-to-cloud connectivity.

DNS Routing Policies

Simple: Single resource, round-robin if multiple values.

Weighted: Distribute traffic by percentage (90/10 split for canary).

Latency-Based: Route to lowest-latency region for the user.

Geolocation: Route based on user's geographic location (compliance, localization).

Failover: Active-passive with health checks.

CDN Best Practices

Cache Headers: Set appropriate Cache-Control headers. Long TTL for static assets, short for dynamic.

Invalidation: Use versioned URLs (style.v2.css) instead of cache invalidation when possible.

Compression: Enable Gzip/Brotli compression for text-based content.

Security: Restrict origin access to CDN only. Use signed URLs for private content.

WAF Rules & Managed Rule Sets

OWASP Core Rule Set: Blocks SQL injection, XSS, LFI, RFI, and command injection. Start with this.

IP Reputation: Block known bad actors, Tor exit nodes, and anonymous proxies.

Rate Limiting: Prevent brute force and credential stuffing attacks.

Bot Control: Challenge or block automated traffic (scrapers, credential stuffers).

Network Firewall Features

Stateful Inspection: Track connection state, auto-allow return traffic.

Domain Filtering: Allow/deny traffic by domain name (not just IP).

IDS/IPS: Detect and block known attack signatures (Suricata rules).

TLS Inspection: Decrypt and inspect encrypted traffic (with caveats).

IAM Best Practices

Least Privilege: Grant minimum permissions needed. Start with none, add as required.

No Root/Owner: Never use root account for daily tasks. Secure with MFA, alert on use.

Roles over Users: Use IAM roles for services and cross-account access. Avoid long-lived access keys.

Federate Identities: Connect enterprise IdP. Single source of truth for users.

Privilege Escalation Prevention

Permission Boundaries: Set maximum permissions a role can ever have (AWS).

SCPs: Service Control Policies limit what entire accounts can do (AWS Organizations).

Deny Policies: Explicitly deny dangerous actions across the organization.

Just-in-Time Access: Grant elevated permissions temporarily with approval workflows.

Zero Trust Architecture

Never trust, always verify

Zero Trust Pillars
üë§
Identity
Verify every user and service. Strong auth, MFA, continuous validation.
üíª
Devices
Validate device health and compliance before granting access.
üåê
Network
Micro-segmentation. No implicit trust based on network location.
üì±
Applications
Secure access to apps regardless of location. API authentication.
üíæ
Data
Classify, encrypt, and control access to data at all times.

Zero Trust Principles

Verify Explicitly: Authenticate and authorize every request based on all available data (identity, location, device, data classification).

Least Privilege: Limit access with just-in-time and just-enough-access. Adaptive policies based on risk.

Assume Breach: Minimize blast radius. Segment access, use end-to-end encryption, verify continuously.

Implementation Steps

1. Strong Identity: Deploy MFA everywhere. Integrate all apps with SSO.

2. Device Trust: Require managed/compliant devices. Mobile device management (MDM).

3. Micro-Segmentation: Service mesh, network policies, security groups per workload.

4. Continuous Monitoring: Log everything. Detect anomalies. Automated response.

Shared Responsibility

Provider Responsibility: Physical security, hypervisor, global infrastructure, managed service internals.

Customer Responsibility: Data classification, access control, encryption keys, network configuration, OS patching (on VMs).

Key Point: Cloud providers are compliant; your deployments may not be. You inherit infrastructure compliance but must configure services correctly.

Compliance Tools

Config Rules: AWS Config, Azure Policy, GCP Org Policies‚Äîcontinuous compliance checking.

Audit Evidence: AWS Audit Manager, Azure Compliance Manager‚Äîautomated evidence collection.

Data Residency: Use region constraints, service control policies, resource location policies.

Encryption: Customer-managed keys (BYOK) for audit control. Document key management procedures.
