Logging Architecture

How logs flow from sources to analysis

Logs are discrete, timestamped records of events. Unlike metrics which are numeric samples, logs capture contextual information‚Äîerror messages, stack traces, user actions, API requests. Centralized logging aggregates these events from distributed systems into a searchable repository.

A modern logging pipeline involves collection agents, processing/enrichment, storage, and visualization. The goal: make it possible to search billions of log lines in seconds to answer questions during incident response.

Structured vs Unstructured Logs

Why format matters for searchability

‚úÖ
Structured Logging
JSON, Key-Value Pairs
Structured logs use consistent, machine-parseable formats like JSON. Every field is explicitly named, making queries fast and reliable.
{ "timestamp": "2024-01-15T10:23:45Z", "level": "ERROR", "service": "payment-api", "trace_id": "abc123", "user_id": "u-456", "message": "Payment failed", "error_code": "CARD_DECLINED", "amount": 99.99, "currency": "USD" }
  • Precise field-level queries
  • Easy aggregation and statistics
  • Consistent schema across services
  • Machine learning ready
  • Better correlation with traces
‚ö†Ô∏è
Unstructured Logging
Plain Text, Free Form
Plain text logs require parsing with regex or grok patterns. Field extraction is brittle and changes when log formats change.
2024-01-15 10:23:45 ERROR [payment-api] Payment failed for user u-456. Card declined. Amount: $99.99 USD. Trace: abc123
  • Human readable at a glance
  • No schema to maintain
  • Legacy system compatibility
  • Regex parsing required
  • Fragile field extraction

Query Languages

Searching and analyzing logs

Each logging platform has its own query language with distinct syntax and capabilities. Mastering these languages is essential for effective log analysis, incident response, and building alerting rules. Below are comprehensive references for the most common platforms.

üîç Elasticsearch KQL / Lucene
Basic Field Match
level: "ERROR" AND service: "payment-api"
Filter by exact field values. Use quotes for strings.
Time Range Filter
@timestamp >= "2024-01-15" AND @timestamp < "2024-01-16"
Query within specific date/time range.
Wildcard Search
message: *timeout* AND NOT service: health-check
Pattern matching with wildcards, exclusions.
Range Queries
response_time: [100 TO 500] AND status: [400 TO 599]
Numeric ranges with inclusive brackets.
Regex Pattern
message: /user-[0-9]+/ AND level: "WARN"
Regular expression matching in fields.
Exists Check
_exists_: error_code AND NOT _exists_: retry_count
Check if field exists in document.
Fuzzy Match
message: "database~2" OR message: "databse"
Approximate matching with edit distance.
Nested Boolean
(level: "ERROR" OR level: "FATAL") AND (service: "api" OR service: "web")
Complex boolean logic with grouping.
üìä LogQL (Grafana Loki)
Label Selector
{app="nginx", env="prod", cluster="us-east"}
Select streams by labels (indexed, fast).
Line Filter
{app="api"} |= "error" != "healthcheck"
Contains (|=) and not contains (!=) filters.
Regex Filter
{app="api"} |~ "user_id=[0-9]+" !~ "test"
Regex match (|~) and not match (!~).
JSON Parser
{app="api"} | json | level="error" | status >= 500
Parse JSON, filter on extracted fields.
Logfmt Parser
{app="api"} | logfmt | duration > 1s | method="POST"
Parse key=value format logs.
Pattern Parser
{app="nginx"} | pattern `<ip> - - <_> "<method> <path>"` | method="POST"
Extract fields using pattern template.
Rate Calculation
rate({app="api"} |= "error" [5m])
Error rate per second over 5 minutes.
Count Over Time
count_over_time({app="api"} |= "error" [1h])
Total error count in last hour.
Sum by Label
sum by (status) (rate({app="api"} | json [5m]))
Aggregate rates grouped by status.
Top K
topk(5, sum by (path) (rate({app="api"} | json [5m])))
Top 5 paths by request rate.
Quantile
quantile_over_time(0.99, {app="api"} | json | unwrap duration [5m]) by (service)
P99 duration by service.
Line Format
{app="api"} | json | line_format "{{.level}} {{.message}}"
Reformat output using Go templates.
üü¢ Splunk SPL
Index Search
index=prod sourcetype=nginx level=ERROR
Search specific index and sourcetype.
Stats Command
index=prod level=ERROR | stats count by host, service
Aggregate counts grouped by fields.
Timechart
index=prod | timechart span=1h count by level
Time-series chart with hourly buckets.
Top Values
index=prod | top 10 uri_path by status
Top 10 paths grouped by status code.
Field Extraction
index=prod | rex field=message "user=(?<username>\w+)"
Extract fields using regex named groups.
Eval Calculation
index=prod | eval duration_ms=duration*1000 | where duration_ms > 1000
Create calculated fields, filter results.
Transaction
index=prod | transaction session_id maxspan=30m
Group events by session within time window.
Lookup Enrichment
index=prod | lookup user_info user_id OUTPUT department, manager
Join with lookup table for enrichment.
Percentiles
index=prod | stats perc50(duration) perc95(duration) perc99(duration) by service
Calculate percentiles by service.
Subsearch
index=prod [search index=alerts | fields host] | stats count by host
Use subsearch results as filter.
Rare Values
index=prod | rare status_code limit=20
Find least common values (anomalies).
Dedup
index=prod | dedup user_id sortby -_time | table user_id, action, _time
Remove duplicates, keep most recent.

Filter & Display

Stats Aggregation

Contextual Logging

Retention Strategies

Balancing cost, compliance, and searchability

Log retention is a tradeoff between cost, compliance requirements, and operational needs. Hot storage enables fast search but is expensive. Archive to cold storage for compliance while keeping recent logs searchable.

Best Practices

Guidelines for effective logging

1

Use Structured Logging

Always emit JSON logs. Include consistent fields: timestamp, level, service, trace_id, message.

2

Include Context

Add request_id, user_id, tenant_id to every log. Enables filtering and correlation.

3

Log at Appropriate Level

DEBUG for development, INFO for operations, ERROR for failures. Don't log sensitive data.

4

Centralize Everything

Ship all logs to central store. Local files are useless in distributed systems.

5

Use UTC Timestamps

Always log in UTC with ISO 8601 format. Avoid timezone confusion across regions.

6

Set Up Retention Policies

Automate log lifecycle: hot ‚Üí warm ‚Üí cold ‚Üí archive. Balance cost and compliance.

7

Monitor Log Pipeline

Alert on dropped logs, parsing failures, ingestion delays. Logs about logs.

8

Create Saved Searches

Pre-build common queries: errors by service, failed logins, slow requests. Ready for incidents.

Anti-Patterns to Avoid

Common logging mistakes

üîê Logging Sensitive Data
Passwords, API keys, credit cards, PII in logs. Security and compliance nightmare.
Use log scrubbing/masking. Never log secrets.
üì¢ Excessive Logging
Logging every function call, loop iteration, or byte. Explodes storage costs.
Log meaningful events. Use sampling for verbose paths.
üîá Insufficient Logging
Only logging errors with no context. "An error occurred" helps no one.
Include relevant context: what, who, when, why.
üìù Inconsistent Format
Different services using different log formats. Some JSON, some plain text.
Standardize on structured format across all services.
üñ•Ô∏è Local-Only Logs
Keeping logs only on the server. Containers die, logs disappear forever.
Ship to central store immediately. Don't rely on local storage.
‚è±Ô∏è Missing Timestamps
Relying on ingestion time or inconsistent time formats. Breaks timeline analysis.
Always include UTC timestamp in ISO 8601 format.
