What is Data Quality?

The foundation of data trust

Beyond "Garbage In, Garbage Out"

Data quality is the degree to which data meets the requirements of its intended use. Poor quality data leads to incorrect insights, flawed ML models, compliance violations, and eroded trust. In a world of AI and analytics, data quality isn't optional‚Äîit's the foundation everything else depends on.

Shift Left: Test Early, Test Often

Modern data quality embraces "shift left" testing‚Äîcatching issues as early as possible in the pipeline rather than discovering them in dashboards or ML predictions. Just as software engineering adopted unit testing, data engineering now embeds quality checks throughout the data lifecycle.

Data Contracts & Trust

Data contracts define expectations between data producers and consumers. They specify what quality guarantees a dataset provides‚Äîschema, freshness, completeness, and more. When combined with automated testing, contracts create accountability and trust across the data ecosystem.

Quality Dimensions

The six pillars of data quality

üéØ
Accuracy
Data correctly represents the real-world entity or event it describes
e.g., Customer age matches their actual age
üì¶
Completeness
All required data is present with no missing values or records
e.g., No NULL email addresses for active users
üîÑ
Consistency
Data is uniform across systems and doesn't contradict itself
e.g., Order total = sum of line items
‚è∞
Timeliness
Data is available when needed and reflects current state
e.g., Dashboard updates within 15 minutes
‚úÖ
Validity
Data conforms to defined formats, types, and ranges
e.g., Email addresses match regex pattern
üîó
Uniqueness
No duplicate records exist where uniqueness is expected
e.g., One record per customer_id

Testing Types

Different tests for different needs

üìã
Schema Tests
Structure validation
Validate that data conforms to expected structure‚Äîcolumn names, data types, nullable constraints.
  • expect_column_to_exist
  • expect_column_values_to_be_of_type
  • expect_table_columns_to_match_set
üìä
Volume Tests
Row count validation
Check that data volume is within expected ranges‚Äîcatches missing data or unexpected duplicates.
  • expect_table_row_count_to_be_between
  • row_count > yesterday * 0.9
  • freshness_check(hours=2)
üéØ
Value Tests
Content validation
Validate actual data values‚Äîranges, patterns, allowed values, and business rules.
  • expect_column_values_to_be_in_set
  • expect_column_values_to_match_regex
  • expect_column_values_to_be_between
üîó
Relationship Tests
Referential integrity
Ensure foreign keys reference valid primary keys‚Äîcritical for joins and dimensional models.
  • expect_column_values_to_be_in_other_table
  • relationships (dbt)
  • referential_integrity_check

Great Expectations

Python-based data validation

Great Expectations Suite Example
Python
import great_expectations as gx

# Create a Data Context
context = gx.get_context()

# Connect to your data source
datasource = context.sources.add_pandas("orders_source")
data_asset = datasource.add_dataframe_asset("orders")
batch_request = data_asset.build_batch_request(dataframe=orders_df)

# Create expectation suite
suite = context.add_expectation_suite("orders_quality_suite")

# Add expectations
validator = context.get_validator(
    batch_request=batch_request,
    expectation_suite_name="orders_quality_suite"
)

# Schema expectations
validator.expect_column_to_exist("order_id")
validator.expect_column_to_exist("customer_id")
validator.expect_column_to_exist("order_total")

# Uniqueness expectations
validator.expect_column_values_to_be_unique("order_id")

# Completeness expectations
validator.expect_column_values_to_not_be_null("customer_id")
validator.expect_column_values_to_not_be_null("order_total")

# Value range expectations
validator.expect_column_values_to_be_between(
    "order_total", min_value=0, max_value=100000
)

# Valid values expectations
validator.expect_column_values_to_be_in_set(
    "status", ["pending", "shipped", "delivered", "cancelled"]
)

# Save and run validation
validator.save_expectation_suite()
results = validator.validate()

# Check results
if results.success:
    print("‚úÖ All quality checks passed!")
else:
    print("‚ùå Quality checks failed")
    for result in results.results:
        if not result.success:
            print(f"  - {result.expectation_config.expectation_type}")

Best Practices

Guidelines for data quality success

‚úì Do's
  • Test at every stage: ingestion, transformation, serving
  • Start with critical tables and expand coverage over time
  • Use data contracts between producers and consumers
  • Alert on failures but avoid alert fatigue
  • Document expectations and share quality reports
  • Version control your test suites alongside code
  • Monitor quality metrics over time for trends
  • Automate quality checks in CI/CD pipelines
‚úó Don'ts
  • Don't test everything‚Äîfocus on business-critical data
  • Avoid hardcoded thresholds without business context
  • Never ignore failing tests‚Äîfix or adjust expectations
  • Don't skip freshness checks on time-sensitive data
  • Avoid running quality checks only in production
  • Don't treat data quality as a one-time project
  • Never blame data producers‚Äîcollaborate on solutions
  • Avoid quality checks that can't be explained to stakeholders
