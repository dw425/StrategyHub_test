Metrics Architecture

How metrics flow from applications to dashboards

Metrics are numeric measurements collected at regular intervals over time. Unlike logs which capture discrete events, metrics provide a continuous view of system state‚ÄîCPU usage, request rates, error counts, and latency percentiles. This time-series data is the backbone of dashboards, alerting, and capacity planning.

A well-designed metrics architecture ensures you can answer questions about system health within seconds. The typical flow involves instrumentation at the application layer, collection by a time-series database, and visualization through dashboards with alerting on anomalies.

Pull vs Push Models

Two approaches to metrics collection

üì•
Pull-Based Model
Prometheus, VictoriaMetrics
The monitoring system actively scrapes metrics from targets at regular intervals. Targets expose an HTTP endpoint (typically /metrics) that returns current metric values.
  • Monitoring system controls scrape schedule
  • Easy to detect when targets are down
  • Simpler firewall rules (outbound only)
  • Native service discovery integration
  • Dominant in cloud-native ecosystems
Prometheus VictoriaMetrics Thanos Cortex
üì§
Push-Based Model
StatsD, InfluxDB, Datadog
Applications actively send metrics to a central collector. The application is responsible for batching and transmitting data at appropriate intervals.
  • Works behind firewalls and NAT
  • Better for short-lived batch jobs
  • Application controls timing
  • Easier initial setup
  • Traditional datacenter approach
StatsD InfluxDB Datadog Agent Graphite

Prometheus Metric Types

The four fundamental metric types

üìà
Counter
Cumulative, monotonically increasing
A counter only increases (or resets to zero on restart). Use counters to track totals: requests, errors, bytes. Always use rate() or increase() to query.
http_requests_total{method="POST", status="200"} 1027 http_requests_total{method="POST", status="500"} 3
Request counts Error totals Bytes transferred
üå°Ô∏è
Gauge
Point-in-time, can go up or down
A gauge represents a value that can increase or decrease. Used for memory usage, temperature, queue depth, concurrent connections.
node_memory_MemAvailable_bytes 2.147483648e+09 node_cpu_temperature_celsius 65.5
Memory usage Temperature Queue depth
üìä
Histogram
Distribution in buckets
Histograms sample observations into configurable buckets. Use histogram_quantile() for percentiles. Buckets are cumulative.
http_request_duration_seconds_bucket{le="0.1"} 24054 http_request_duration_seconds_bucket{le="0.5"} 33444 http_request_duration_seconds_bucket{le="+Inf"} 34567
Request latency P50/P95/P99 Response sizes
üìâ
Summary
Pre-calculated quantiles
Summaries calculate streaming œÜ-quantiles on the client. Unlike histograms, quantiles cannot be aggregated across instances.
go_gc_duration_seconds{quantile="0.5"} 0.000107 go_gc_duration_seconds{quantile="0.99"} 0.000706 go_gc_duration_seconds_count 12345
GC duration Single instance Accurate quantiles

Cardinality Management

The #1 cause of metrics system issues

Cardinality refers to the number of unique time series. Each unique combination of metric name and label values creates a new time series. High cardinality‚Äîcaused by labels with many possible values (user IDs, request IDs)‚Äîcan overwhelm your monitoring system.

üö´ High Cardinality vs ‚úÖ Low Cardinality
‚ùå Avoid: High Cardinality
http_requests_total{ method="GET", user_id="u-12345", # Millions! request_id="req-abc", # Unique! status="200" } // Creates millions of series!
‚úì Better: Bounded Labels
http_requests_total{ method="GET", user_type="premium", # ~3 values status_class="2xx", # 5 values region="us-east-1" } // ~1,500 series (manageable)

SLIs, SLOs & Error Budgets

Metrics-driven reliability targets

SLI (Service Level Indicator) is a quantitative measure of service behavior. SLO (Service Level Objective) is a target value for an SLI. Error Budget is the allowed unreliability (100% - SLO).

Best Practices

Design patterns for effective metrics

1

Use Consistent Naming

Follow namespace_subsystem_name_unit. Include units: _seconds, _bytes, _total.

2

Control Cardinality

Avoid unbounded labels (user IDs, request IDs). Keep per-metric cardinality under 10K.

3

Prefer Histograms for Latency

Histograms allow aggregation across instances. Choose buckets based on SLO thresholds.

4

Instrument Golden Signals

Every service should expose latency, traffic, errors, saturation as minimum baseline.

5

Use Recording Rules

Pre-compute expensive queries. Improves dashboard performance and enables consistent alerting.

6

Add Metadata Labels

Include useful context: team, env, version for filtering and aggregation.

7

Document Metrics

Use HELP text and TYPE declarations. Maintain a metrics catalog for discoverability.

8

Monitor the Monitoring

Track Prometheus memory, scrape duration, failed scrapes. Alert on monitoring system health.

Anti-Patterns to Avoid

Common mistakes that hurt metrics systems

üí• High-Cardinality Labels
Using user_id, request_id, or email as labels creates millions of time series, causing OOM crashes.
Use bounded labels like user_type, customer_tier, or region.
üìä Querying Raw Counters
Displaying counter values directly is meaningless. The raw number always increases.
Always use rate() or increase() with counters.
‚è∞ Too Short Scrape Intervals
1-second scrape intervals waste resources. Most metrics don't change that fast.
Use 15-30 second intervals for most targets.
üîÄ Inconsistent Label Names
Using "svc" in one metric and "service" in another breaks correlation.
Establish and enforce naming conventions organization-wide.
üìà Averaging Percentiles
Taking avg() of P99 values gives mathematically meaningless numbers.
Use histogram_quantile() on aggregated buckets.
üîï Alert Fatigue
Alerting on every metric spike. Teams ignore alerts, miss real incidents.
Alert on symptoms (user impact), not causes.

Agent This

AI-powered metrics optimization

The MetricsEngineer agent automates the complex task of metrics design and query optimization by analyzing application telemetry patterns, identifying cardinality issues, and generating efficient PromQL queries. It continuously monitors dashboard performance and suggests recording rules to pre-compute expensive queries.

The agent integrates with Prometheus and Grafana APIs to analyze metric schemas, builds dependency graphs between metrics and dashboards, and uses statistical methods to identify anomalous patterns. It provides what-if analysis for cardinality impact and alerts when new labels or metrics risk exploding series counts.
