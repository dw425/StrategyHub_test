Understanding RPA

What RPA is, when to use it, and when to avoid it

What is Robotic Process Automation?

RPA automates tasks by mimicking human interactions with software. Rather than connecting through APIs (the approach used by iPaaS and low-code platforms), RPA bots interact with applications the same way humans do‚Äîclicking buttons, typing text, reading screens, moving files. This "surface-level" automation works with any application that has a user interface, regardless of whether it exposes APIs.

The technology uses screen scraping, image recognition, and UI element identification to "see" what's on screen and interact with it. Modern RPA platforms combine these techniques with AI (Computer Vision, NLP, Document Understanding) to handle unstructured content and make decisions that previously required human judgment.

RPA fills the gap where APIs don't exist. Legacy mainframes, desktop applications from the 1990s, web applications that block programmatic access‚ÄîRPA can automate these when nothing else can. But this strength is also its limitation: UI automation is inherently fragile, slower than API integration, and requires ongoing maintenance.

When to Use RPA vs. API Integration

Use RPA when: No API exists, the application is legacy/mainframe, you need to automate across multiple disconnected systems, the process involves desktop applications, or you need a quick win before a larger transformation project.

Prefer API integration when: APIs are available, you need high-volume/high-speed processing, the integration is mission-critical, or you're building for long-term scalability. API integrations are faster, more reliable, and easier to maintain.

The hybrid approach often wins: Many real-world processes combine API calls for modern systems with RPA for legacy touchpoints. A single automation might pull data via API from Salesforce, process it in Excel using RPA, and update a mainframe terminal session‚Äîall orchestrated together.

Market Landscape

The competitive landscape of RPA platforms

The Big Three (Plus Microsoft)

UiPath leads in market share and mindshare, with the most extensive feature set and largest community. They've expanded beyond RPA into full "automation platform" territory with process mining, test automation, and AI capabilities.

Automation Anywhere pioneered cloud-native RPA and has strong enterprise credentials. Their recent focus on generative AI integration positions them for the next wave of intelligent automation.

Blue Prism (now part of SS&C) originated enterprise RPA and maintains a reputation for governance and security. Popular in heavily regulated industries like financial services.

Microsoft Power Automate Desktop disrupted the market by including RPA capabilities free with Windows 11 and M365. Less feature-rich than the dedicated vendors but deeply integrated with the Microsoft ecosystem.

Understanding RPA Costs

Licenses are just the beginning. Implementation, training, infrastructure, and ongoing maintenance often exceed software costs. Budget 2-3x license fees for total program cost.

Bot maintenance is ongoing. Applications change, breaking automations. Plan for 20-30% of development effort annually for maintenance and fixes.

Hidden costs include: VDI infrastructure for unattended bots, exception handling development, monitoring tools, and the CoE team to govern the program.

Best Practices

Lessons from successful RPA programs

Do These Things
  • ‚úì Start with process standardization before automation‚Äîfix the process first.
  • ‚úì Build a Center of Excellence to govern, support, and scale the program.
  • ‚úì Design for exception handling‚Äîevery automation will encounter unexpected scenarios.
  • ‚úì Create reusable components and libraries to accelerate future development.
  • ‚úì Monitor bot performance and business outcomes, not just technical metrics.
  • ‚úì Plan for maintenance‚Äîapplications change, and bots break.
Avoid These Mistakes
  • ‚úó Automating broken or unstable processes‚Äîgarbage in, garbage out faster.
  • ‚úó Choosing RPA when an API integration would be more reliable and maintainable.
  • ‚úó Underestimating maintenance effort‚Äîplan for 25% of dev time ongoing.
  • ‚úó Deploying without proper error handling and logging.
  • ‚úó Letting citizen developers build without governance or standards.
  • ‚úó Expecting "set and forget"‚ÄîRPA requires continuous attention.
