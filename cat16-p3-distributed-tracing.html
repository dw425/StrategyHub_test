Tracing Architecture

How traces flow through distributed systems

Distributed tracing captures the journey of a request across service boundaries. Each service creates a "span" recording its work, and these spans are linked by a shared trace ID. Together, spans form a tree showing the complete request path, timing, and relationships.

Context propagation is the mechanism that passes trace identifiers between services via HTTP headers, message queues, or RPC metadata. This enables correlation of work done by different services into a single, unified trace.

Core Concepts

Fundamental building blocks of distributed tracing

üîó
Trace
A trace represents the entire journey of a request through all services. It's identified by a unique trace ID that's propagated across service boundaries.
  • üÜî Unique 128-bit trace ID
  • üå≥ Tree structure of spans
  • ‚è±Ô∏è End-to-end latency
  • üîç Cross-service correlation
üì¶
Span
A span represents a single unit of work within a trace. It has a name, start time, duration, and can contain attributes, events, and links to other spans.
  • üÜî Unique span ID
  • üëÜ Parent span reference
  • üè∑Ô∏è Attributes (key-value)
  • üìù Events (timestamped logs)
üîÑ
Context
Context carries trace identifiers across service boundaries. It's propagated via HTTP headers, message metadata, or RPC frameworks.
  • üì® HTTP headers (W3C)
  • üì¨ Message queue metadata
  • üîå gRPC metadata
  • üß≥ Baggage items

Span Anatomy

Structure and fields of a span

üì¶ Span Data Model

trace_id
string (128-bit hex)
Globally unique identifier linking all spans in a trace. Example: "4bf92f3577b34da6a3ce929d0e0e4736"
span_id
string (64-bit hex)
Unique identifier for this span within the trace. Example: "00f067aa0ba902b7"
parent_span_id
string (optional)
ID of the parent span. Null for root spans. Creates the tree structure.
name
string
Operation name. Examples: "HTTP GET /api/users", "SELECT users", "kafka.produce"
kind
enum
Span type: CLIENT, SERVER, PRODUCER, CONSUMER, INTERNAL
start_time / end_time
timestamp (nanoseconds)
When the operation started and ended. Duration = end_time - start_time
attributes
map<string, any>
Key-value metadata: http.method, http.status_code, db.system, user.id, etc.
events
list<Event>
Timestamped annotations within the span. Useful for logging exceptions or milestones.
status
enum + message
OK, ERROR, or UNSET with optional error message.
links
list<Link>
References to related spans in other traces (batch processing, fan-out).

Context Propagation

How trace context flows between services

Context propagation is the mechanism that carries trace identifiers across process boundaries. Without proper propagation, spans from different services cannot be correlated into a single trace. The W3C Trace Context standard provides interoperability between tracing systems.

üåê
W3C Trace Context (Standard)
The W3C standard for trace context propagation. Supported by all major tracing systems and ensures interoperability.
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 tracestate: vendor1=value1,vendor2=value2
üÖ±Ô∏è
B3 Propagation (Zipkin)
Zipkin's original format. Still widely used. Available in single-header or multi-header variants.
X-B3-TraceId: 4bf92f3577b34da6a3ce929d0e0e4736 X-B3-SpanId: 00f067aa0ba902b7 X-B3-ParentSpanId: 00f067aa0ba902b6 X-B3-Sampled: 1
üü°
Jaeger Propagation
Jaeger's native format using uber-trace-id header. Compact single-header format.
uber-trace-id: 4bf92f3577b34da6:00f067aa0ba902b7:00f067aa0ba902b6:1
üß≥
Baggage (Custom Context)
Carry custom key-value pairs across service boundaries. Useful for tenant ID, feature flags, or request metadata.
baggage: userId=alice,tenantId=acme,featureFlag=newUI

Sampling Strategies

Managing trace volume and cost

At scale, collecting every trace is prohibitively expensive. Sampling strategies help reduce volume while preserving important traces. Head-based sampling decides early, while tail-based sampling can make intelligent decisions after seeing the complete trace.

üé≤
Probabilistic
Random sampling at a fixed rate. Simple but may miss important traces.
Rate: 1-10%
‚è±Ô∏è
Rate Limiting
Fixed number of traces per second. Prevents cost spikes during traffic surges.
e.g., 100 traces/sec
üéØ
Adaptive
Adjusts rate based on traffic patterns. Higher rate for low-volume endpoints.
Dynamic
üêå
Latency-Based
Always sample slow requests. Captures performance issues regardless of rate.
e.g., > 2s latency
üö®
Error-Based
Always sample traces with errors. Never miss failed requests for debugging.
100% errors
üîö
Tail-Based
Decide after trace completes. Can sample based on duration, errors, or attributes.
Intelligent

Best Practices

Guidelines for effective distributed tracing

1

Use OpenTelemetry

Instrument once with OTel, export anywhere. Vendor-neutral, future-proof, industry standard.

2

Propagate W3C Trace Context

Use W3C traceparent/tracestate headers. Ensures interoperability between all services and vendors.

3

Add Meaningful Attributes

Include user.id, order.id, tenant.id. Makes traces searchable and useful for debugging.

4

Name Spans Descriptively

Use operation names like "HTTP GET /api/orders" or "SELECT users". Avoid generic names.

5

Record Errors Properly

Set span status to ERROR, add exception events with stack traces. Essential for debugging.

6

Use Tail-Based Sampling

Sample based on outcomes (errors, high latency). Never miss important traces.

7

Correlate with Logs

Include trace_id in all log messages. Click from trace to logs and vice versa.

8

Monitor Trace Pipeline

Track dropped spans, collector lag, storage costs. Ensure tracing itself is healthy.

Anti-Patterns to Avoid

Common distributed tracing mistakes

üîó Breaking Context Propagation
Not passing trace headers through async boundaries, message queues, or thread pools. Creates orphan spans.
Always extract/inject context at service boundaries. Use instrumented clients.
üìä 100% Sampling in Production
Collecting every trace at high traffic. Explodes storage costs and can impact performance.
Use intelligent sampling. Sample all errors/slow requests, sample others probabilistically.
üè∑Ô∏è High-Cardinality Attributes
Adding unique IDs like request_id or session_id as attributes. Explodes index size.
Use bounded attributes. Store unique IDs in events, not span attributes.
üìù Generic Span Names
Using names like "execute", "process", or "handle". Impossible to understand trace structure.
Use descriptive names: "HTTP POST /checkout", "redis.get user:123".
üîí Sensitive Data in Traces
Including passwords, tokens, or PII in span attributes. Security and compliance risk.
Scrub sensitive data before export. Use allowlists for attributes.
üèùÔ∏è Isolated Tracing
Not correlating traces with logs and metrics. Missing the full observability picture.
Include trace_id in logs. Use exemplars to link metrics to traces.
