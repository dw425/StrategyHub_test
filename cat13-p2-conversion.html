ğŸ” Key Transformations Explained

Lines 1-3
Procedure â†’ Table Function
Oracle's PROCEDURE with SYS_REFCURSOR OUT parameter becomes a Databricks FUNCTION RETURNS TABLE. This is a structural transformationâ€”the procedure's cursor-based output is replaced with a direct table return, which is more idiomatic for Spark SQL and enables better query optimization.
Line 2
NUMBER â†’ INT
Oracle's NUMBER type maps to Databricks INT for whole numbers. The conversion engine analyzes usage context to choose the appropriate numeric type (INT, BIGINT, DECIMAL, DOUBLE).
Line 8
NVL â†’ COALESCE
Oracle's proprietary NVL() function converts to ANSI-standard COALESCE(). COALESCE is more portable and supports multiple fallback values, making the code more maintainable.
Line 9
TO_CHAR â†’ DATE_FORMAT
Date formatting syntax changes from Oracle's TO_CHAR(date, 'YYYY-MM') to Spark's DATE_FORMAT(date, 'yyyy-MM'). Note the lowercase format specifiersâ€”the engine handles these subtle differences automatically.
Line 11
EXTRACT â†’ YEAR()
While EXTRACT is ANSI SQL, Databricks' native YEAR() function is more performant and commonly used. The optimizer recognizes this pattern and applies the idiomatic replacement.
Line 5
OPEN CURSOR â†’ Removed
The OPEN p_result FOR cursor operation is eliminated entirely. Table functions in Databricks return results directly without cursor management, simplifying the code and improving performance.

Conversion Rules

Pattern-based transformation categories

Conversion rules are organized into three categories based on transformation complexity. Syntax rules handle direct text substitutions, function rules manage parameter reordering and semantic changes, and procedural rules restructure control flow and program architecture.

Each rule is tested against a corpus of 10,000+ real-world code samples to ensure accuracy. Rules include both the transformation logic and regression tests to prevent breaking changes when the rule library is updated.

Edge Cases & Manual Review

Patterns requiring human intervention

Not all code can be automatically converted. The conversion engine identifies patterns that are too complex, ambiguous, or platform-specific for automated handling. These are flagged for manual review by experienced engineers.

Edge cases are prioritized by severity (impact on migration success) and frequency (how many objects are affected). High-severity items are addressed first to unblock dependent conversions.
