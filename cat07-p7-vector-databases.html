Understanding Vector Databases

Specialized storage for semantic similarity search

Why Vector Databases?

Vector databases are purpose-built to store and query high-dimensional vectors (embeddings). Unlike traditional databases optimized for exact matches on structured data, vector databases use approximate nearest neighbor (ANN) algorithms to find semantically similar content in milliseconds‚Äîeven across billions of vectors.

At their core, vector databases solve the similarity search problem: given a query vector (like an embedded question), find the K most similar vectors (like document chunks). This enables RAG systems to retrieve relevant context, recommendation engines to find similar items, and search systems to understand user intent beyond keywords.

Key capabilities include metadata filtering (narrow results by attributes before vector search), hybrid search (combining dense vectors with sparse keyword matching), and namespaces/collections (multi-tenancy for different use cases or customers). Modern vector databases also support real-time updates, allowing you to add, update, or delete vectors without full re-indexing.

Embeddings are dense numerical representations that capture semantic meaning. When you embed "How do I reset my password?" and "Password recovery steps", they produce similar vectors‚Äîeven though they share few words. This semantic similarity powers modern search and RAG.

Popular embedding models include OpenAI text-embedding-3 (1536 or 3072 dimensions), Cohere embed-v3 (1024 dimensions), and open-source options like BGE, E5, and GTE. Dimension count affects both accuracy and storage/compute costs. Higher dimensions generally mean better accuracy but slower queries and more storage.

Index Algorithms

How vector databases find similar vectors fast

üåê
HNSW
Hierarchical Navigable Small World
Graph-based algorithm with multiple layers. Excellent recall and query speed. Most popular choice for production.
99%+ Recall
~1ms Query
üì¶
IVF
Inverted File Index
Clusters vectors into buckets, searches only relevant clusters. Good balance of build time and query speed.
95%+ Recall
~5ms Query
üóúÔ∏è
PQ
Product Quantization
Compresses vectors to reduce memory footprint. Often combined with IVF for large-scale deployments.
90%+ Recall
8x Compression
üéØ
Flat
Brute Force (Exact)
Compares query against every vector. 100% accurate but slow at scale. Good for small datasets or evaluation.
100% Recall
O(n) Speed

HNSW is the most popular algorithm in production vector databases. It builds a multi-layer graph where each node connects to its nearest neighbors. Queries start at the top layer (sparse, long-range connections) and descend to lower layers (dense, local connections), efficiently navigating to the nearest neighbors. Trade-offs are controlled by M (connections per node) and efConstruction (build-time quality).

IVF-PQ combines clustering with compression for billion-scale deployments. Vectors are clustered during indexing, and only relevant clusters are searched at query time. Product Quantization compresses vectors by splitting dimensions into subspaces and quantizing each independently‚Äîachieving 4-8x memory reduction with acceptable recall loss.

Architecture Patterns

Deployment and scaling strategies

üéØ
Single Node
Simple deployment for development, testing, and small production workloads up to ~10M vectors.
  • Fastest to deploy and iterate
  • No distributed complexity
  • Limited by single machine memory
  • Best for: POCs, small apps
üìä
Sharded Cluster
Distribute vectors across multiple nodes for horizontal scaling to billions of vectors.
  • Horizontal scaling
  • Query parallelization
  • Consistent hashing for routing
  • Best for: High-scale production
üîÑ
Replicated
Multiple copies of data for high availability and read scaling. Essential for production.
  • Fault tolerance
  • Read throughput scaling
  • Leader-follower or multi-master
  • Best for: HA requirements

Most vector databases support sharding (distributing vectors across nodes) and replication (copying vectors for availability). For production, use at least 3 replicas with automatic failover. Pinecone handles this automatically; self-hosted options require explicit configuration.

Consider separation of indexing and serving at scale: index new vectors on dedicated nodes, then replicate to serving nodes. This prevents indexing load from affecting query latency. Milvus and Vespa have built-in support for this pattern.

Performance Optimization

Tuning for latency, throughput, and cost

üóúÔ∏è
Quantization
Compress vectors to reduce memory and speed up search with minimal accuracy loss.
Use scalar quantization (SQ8) for 4x compression with ~1% recall loss
üéöÔ∏è
Index Tuning
Adjust HNSW parameters M and efConstruction to balance build time, memory, and recall.
Start with M=16, efConstruction=200, increase for higher recall
üîç
Search Parameters
Tune efSearch at query time to trade latency for accuracy based on use case.
Use efSearch=50 for low latency, efSearch=200+ for high recall
üì¶
Batching
Batch upserts and queries to amortize network overhead and improve throughput.
Batch 100-1000 vectors per upsert, 10-50 queries per batch

Quantization is the highest-impact optimization for cost reduction. Scalar quantization (SQ8) reduces memory 4x by converting float32 to int8. Binary quantization (BQ) achieves 32x compression but with larger recall loss. Most databases support multiple quantization levels‚Äîtest on your data to find the right trade-off.

Dimension reduction is another option: Matryoshka embeddings (like OpenAI's text-embedding-3) allow truncating vectors to smaller dimensions. Reducing from 3072 to 1024 dimensions cuts storage 3x with minimal quality loss for many use cases. Always benchmark on your specific queries.

Best Practices

Guidelines for production vector databases

‚úì Do
  • Test embedding models on your specific domain data
  • Use hybrid search (vectors + keywords) for best results
  • Store useful metadata alongside vectors for filtering
  • Implement proper chunking strategy for documents
  • Monitor recall and latency in production
  • Use namespaces/collections for multi-tenancy
  • Benchmark quantization impact on your queries
  • Plan for index rebuilds when changing parameters
‚úó Don't
  • Assume all embedding models perform equally
  • Use flat/brute-force search in production at scale
  • Ignore the cost of high-dimensional embeddings
  • Skip metadata filtering when you have structured attributes
  • Embed entire documents‚Äîuse smart chunking
  • Forget to handle embedding model version changes
  • Mix embeddings from different models in one index
  • Assume managed service costs scale linearly

# vector_agent.py - Vector Database Agent
from crewai import Agent, Task
import openai
from pinecone import Pinecone

class VectorOpsAgent:
    def __init__(self, index_name: str):
        self.pc = Pinecone()
        self.index = self.pc.Index(index_name)
        self.embed_model = "text-embedding-3-small"
    
    def embed_and_upsert(self, docs: list, namespace: str):
        """Embed documents and upsert to vector DB."""
        chunks = self._chunk_documents(docs)
        
        for batch in self._batches(chunks, size=100):
            embeddings = openai.embeddings.create(
                model=self.embed_model,
                input=[c["text"] for c in batch]
            )
            
            vectors = [{
                "id": c["id"],
                "values": e.embedding,
                "metadata": c["metadata"]
            } for c, e in zip(batch, embeddings.data)]
            
            self.index.upsert(vectors, namespace=namespace)
    
    def hybrid_search(self, query: str, filter: dict, k: int = 10):
        """Hybrid search with vector + metadata filtering."""
        query_vec = self._embed(query)
        
        results = self.index.query(
            vector=query_vec,
            filter=filter,
            top_k=k,
            include_metadata=True
        )
        return results.matches

# Agent definition
vector_agent = Agent(
    role="Vector Database Specialist",
    goal="Optimize semantic search and retrieval",
    tools=[EmbedTool(), UpsertTool(), SearchTool(), OptimizeTool()]
)
