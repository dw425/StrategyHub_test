Tactical DDD

Building blocks for domain modeling

üÜîEntity
Object with unique identity that persists over time. Two entities with same attributes but different IDs are different.
üíéValue Object
Immutable object defined by attributes, not identity. Two VOs with same attributes are equal.
üéÅAggregate
Cluster of entities/VOs with a root entity. Unit of consistency and transaction boundaries.
üíæRepository
Abstraction for retrieving and storing aggregates. Collection-like interface hiding persistence.
üì£Domain Event
Record of something that happened (OrderPlaced). Used to communicate between aggregates.
‚öôÔ∏èDomain Service
Stateless operations that don't belong to an entity. Operations spanning multiple aggregates.

Aggregate Design Rules

  • Small aggregates: Prefer smaller, focused aggregates
  • Reference by ID: Aggregates reference others by ID only
  • One transaction: Update one aggregate per transaction
  • Protect invariants: All changes go through the root

Value Object Benefits

  • Immutable: No side effects, safe to share
  • Self-validating: Invalid state is unrepresentable
  • Expressive: Money is better than decimal
  • Replaceable: Create new instead of mutating

Understanding Domain-Driven Design

Deep dive into the philosophy and patterns

üéØ

What is Domain-Driven Design?

Eric Evans, 2003 "Blue Book"

What it is: DDD is an approach to software development that centers the design around the core business domain. It's not a framework or technology‚Äîit's a way of thinking that aligns software models with business reality. The goal is to create software that domain experts can understand and validate, reducing the gap between business concepts and code.

Core Philosophy

  • Domain First: Understand the business before writing code
  • Model as Conversation: Code reflects how experts talk about the domain
  • Bounded Contexts: Different parts of business = different models
  • Continuous Learning: Model evolves as understanding deepens
  • Focus Investment: Not everything needs DDD‚Äîfocus on core domain

The Two Levels

  • Strategic DDD: How to organize the problem space (bounded contexts, context maps, ubiquitous language)
  • Tactical DDD: How to model within a context (entities, value objects, aggregates, repositories)
  • Start Strategic: Always begin with strategic patterns‚Äîthey have the highest ROI
  • Tactical is Optional: Use tactical patterns only in core domain where complexity warrants it
‚úì Business-tech alignment, reduced complexity, maintainable systems, enables microservices, better team communication
‚úó Steep learning curve, requires domain expert access, overkill for simple domains, upfront investment

Bounded Context (The Most Important Pattern)

Strategic DDD

What it is: A bounded context is a boundary within which a particular domain model applies. The same concept (like "Customer") can mean different things in different contexts‚Äîand that's okay. In Sales, a Customer has a sales rep and pipeline stage. In Shipping, a Customer has an address and delivery preferences. These are different models for the same real-world entity.

Why This Matters

Without bounded contexts, you end up with a "God Object"‚Äîa single Customer class with 100 fields trying to serve every use case. This leads to tight coupling, merge conflicts, and a model nobody understands. Bounded contexts allow each team to evolve their model independently.

üéÅ

Aggregates (Transaction Boundaries)

Tactical DDD

What it is: An aggregate is a cluster of domain objects (entities and value objects) that are treated as a single unit for data changes. Every aggregate has a "root" entity‚Äîthe only object external code can reference. All modifications go through the root, which enforces invariants (business rules that must always be true).

When to Use DDD

Decision framework for your project

üó∫Ô∏è

Use Strategic DDD When...

  • ‚úì Multiple teams working on same system
  • ‚úì Breaking apart a monolith
  • ‚úì Designing microservices boundaries
  • ‚úì Integration between different systems
  • ‚úì Need shared vocabulary with business
  • ‚úì Almost always‚Äîlow cost, high value
Example: Any team larger than 3-4 people, any system with multiple bounded contexts, any microservices architecture.
üîß

Use Tactical DDD When...

  • ‚úì Complex business rules and invariants
  • ‚úì Domain experts available to collaborate
  • ‚úì Core domain (competitive advantage)
  • ‚úì Long-lived system (5+ years)
  • ‚úì Behavior-rich domain (not just data)
  • ‚úì Team willing to invest in learning
Example: Insurance claim processing, trading systems, healthcare workflows‚Äîwhere business rules ARE the product.
‚è≠Ô∏è

Skip Strategic DDD When...

  • ‚úó Solo developer, single context
  • ‚úó Throwaway prototype
  • ‚úó No cross-team communication needed
Reality: Strategic DDD is almost always worth it. Even solo projects benefit from thinking about bounded contexts.
‚è≠Ô∏è

Skip Tactical DDD When...

  • ‚úó Simple CRUD operations
  • ‚úó Supporting/generic subdomains
  • ‚úó No domain expert access
  • ‚úó Short-lived or disposable code
  • ‚úó Data-centric, not behavior-centric
Example: Admin panels, settings pages, reporting dashboards‚Äîuse simple CRUD and save DDD for the core.

Quick Decision Tree

No, simple CRUD ‚Üí Simple architecture

Evolution of Software Design

From data-centric to domain-centric

1

Database-Centric

Design starts with ERD. Business logic in stored procedures. Data model drives everything.

2

Layered/N-Tier

Separate UI, business, data layers. But often just anemic models‚ÄîDTOs with getters/setters.

3

Service-Oriented

Extract services, but boundaries often technical not business. "Customer Service" serves everyone.

4

Domain-Driven

Start with domain understanding. Model reflects business reality. Boundaries = business boundaries.

Best Practices

Guidelines for effective DDD

‚úì Do
  • ‚úì Start with strategic DDD‚Äîunderstand bounded contexts first
  • ‚úì Invest in ubiquitous language‚Äîit's the foundation
  • ‚úì Focus tactical DDD on the core domain
  • ‚úì Keep aggregates small (1-3 entities typical)
  • ‚úì Use value objects liberally
  • ‚úì Pair with domain experts regularly
‚úó Don't
  • ‚úó Apply tactical patterns to every bounded context
  • ‚úó Create anemic domain models (getters/setters only)
  • ‚úó Make aggregates too large
  • ‚úó Reference aggregates directly (use IDs)
  • ‚úó Skip the shared language work
  • ‚úó Treat DDD as just a folder structure
