What is Palantir?

The operational data company

The Enterprise Operating System

Palantir builds software platforms that serve as central operating systems for organizations. Unlike traditional data platforms that focus on storage and analytics, Palantir's platforms are designed to integrate data, decisions, and operations at scale. The key differentiator is the Ontology‚Äîa semantic layer that creates a "digital twin" of your organization, connecting raw data to real-world entities like customers, products, facilities, and processes.

Founded for Defense, Expanded to Enterprise

Founded in 2003 by Peter Thiel, Alex Karp, and others with seed funding from In-Q-Tel (the CIA's venture arm), Palantir originally built intelligence software for defense and counterterrorism. The name comes from the "seeing-stones" in Tolkien's Lord of the Rings. Today, Palantir serves both government (55% of revenue) and commercial customers (45%) across healthcare, manufacturing, financial services, and more. Notable customers include Airbus, Ferrari, Merck, the U.S. Department of Defense, and the NHS.

Four Principal Platforms

Palantir has built four interconnected platforms: Gotham (defense/intelligence), Foundry (commercial operations), Apollo (software deployment), and AIP (AI Platform). These platforms share the same Ontology foundation and are backed by Apollo's continuous deployment capabilities, enabling zero-downtime updates across any environment‚Äîfrom public cloud to air-gapped classified networks.

The Ontology

Palantir's core competitive advantage

Digital Twin of Your Organization

The Ontology is Palantir's most important differentiator. Instead of tables and columns, you work with real-world entities like "Employee," "Aircraft," "Purchase Order," or "Field Service Ticket"‚Äîalong with the relationships and business logic that govern them. As CTO Shyam Sankar stated: "Our advantage comes down to Ontology. It's really an advantage on the AI demand side."

Semantic + Kinetic Elements

The Ontology contains both semantic elements (objects, properties, links) that define what things are, and kinetic elements (actions, functions, dynamic security) that define what you can do with them. This allows the Ontology to not just describe your business‚Äîbut to actually operate it, with built-in governance and audit trails.

Object Types
Real-world entities
Define entities like Customer, Equipment, Transaction, or Threat Actor. Each object type maps to backing datasets but abstracts away the technical details.
  • Map datasets to business concepts
  • Rich metadata and descriptions
  • Primary keys and indexing
Properties
Attributes and characteristics
Properties define the characteristics of objects‚Äîlike a Customer's name, email, or lifetime value. Properties can be shared across multiple object types.
  • Shared properties across types
  • Derived/computed properties
  • Time series and versioning
Link Types
Relationships between entities
Links define relationships: Customer ‚Üí Order, Supplier ‚Üí Shipment ‚Üí Warehouse. Unlike foreign keys, links are first-class citizens with their own properties.
  • One-to-many, many-to-many
  • Bidirectional navigation
  • Link properties (strength, date)
Action Types
Controlled modifications
Actions define how objects can be modified‚Äîwith validation rules, approval workflows, and audit trails. All writes must go through Actions, preventing uncontrolled changes.
  • Validation rules and constraints
  • Multi-step approval workflows
  • Complete audit history
Functions
Executable business logic
Functions are code-based logic that operates on the Ontology. They can take objects as input, compute derived values, and power real-time operational applications.
  • TypeScript/Python authoring
  • Ontology-native inputs/outputs
  • Used across apps and actions
Interfaces
Object type polymorphism
Interfaces describe the shape of object types, enabling polymorphism. Multiple object types can implement the same interface for consistent modeling and interaction.
  • Shared shape definitions
  • Cross-type operations
  • Consistent API patterns
Ontology SDK Example
TypeScript
// Query objects using the Ontology SDK
import { Ontology } from "@palantir/ontology-sdk";

// Get all customers in a region
const customers = await Ontology.Customer
    .where(c => c.region.eq("EMEA"))
    .where(c => c.status.eq("Active"))
    .fetchPage();

// Navigate links to related orders
for (const customer of customers.data) {
    const orders = await customer.orders.fetchPage();
    console.log(`${customer.name}: ${orders.data.length} orders`);
}

// Execute an action with validation
await Ontology.Actions.createOrder({
    customerId: customer.id,
    products: ["SKU-001", "SKU-002"],
    priority: "HIGH"
}); // Automatically validated and audited

Palantir vs Databricks

Two different philosophies for data platforms

Fundamentally Different Approaches

While both Palantir and Databricks are leading data platforms, they represent fundamentally different philosophies. Databricks is a lakehouse platform‚Äîfocused on unifying data lakes and warehouses with best-in-class compute (Spark) and storage (Delta Lake). Palantir is an operational platform‚Äîfocused on creating a semantic layer (Ontology) that turns data into actionable business objects that power real-world decisions and workflows.

Data-Centric vs. Operations-Centric

Databricks excels at data engineering, analytics, and ML/AI workloads where the goal is transforming and analyzing data at scale. Palantir excels at operational use cases where the goal is empowering decision-makers (often non-technical) to take actions based on data. Databricks users typically write code; Palantir users often work through low-code applications built on the Ontology.

When to Choose Each

Choose Databricks when you need: massive-scale data processing, a unified lakehouse for analytics and ML, strong SQL/BI capabilities, or deep Spark expertise. Choose Palantir when you need: operational decision-making applications, a semantic layer (digital twin) of your business, tight integration between data and real-world actions, or deployment to air-gapped/classified environments.

Migration: Palantir ‚Üí Databricks

Strategy for migrating backend ETL and data pipelines

What Migrates vs. What Gets Left Behind

When migrating from Palantir Foundry to Databricks, your data and ETL logic can migrate‚Äîbut the Ontology, Actions, and AIP must be rebuilt or replaced. Palantir's transforms are often written in PySpark/SQL which ports directly to Databricks. However, the semantic layer (Ontology) and operational applications (Workshop) have no direct equivalent and require architectural redesign. This is the key trade-off: you gain open formats and ecosystem flexibility, but lose the integrated operational layer.

Migration Complexity by Layer

Low complexity: Raw data extraction, Spark/SQL transforms, basic pipelines. Medium complexity: Data Connection connectors, scheduling, lineage recreation. High complexity: Ontology object models, Actions/Functions, Workshop applications, AIP workflows. The Ontology is typically the hardest part‚Äîit encodes years of business logic and relationships that must be re-implemented in application code or BI tools.

Recommended Approach

Migrate in phases: (1) Extract raw data to cloud storage, (2) Port transform code to Databricks notebooks/jobs, (3) Rebuild data models in Delta Lake + Unity Catalog, (4) Replace Ontology with semantic layer tools or custom APIs, (5) Rebuild operational apps with partner tools or custom development. Prioritize based on business criticality and complexity assessment.

‚úì Migration Do's
  • Document ALL Ontology objects, links, and Actions before starting‚Äîthis is your business logic
  • Export transform code from Code Repositories‚Äîit's mostly standard PySpark
  • Use Foundry's data export APIs to extract datasets in parallel
  • Start with data layer migration before attempting app rebuilds
  • Leverage dbt for semantic layer to partially replace Ontology concepts
  • Plan for extended timeline‚ÄîOntology rebuild is always underestimated
  • Consider hybrid approach during transition (both platforms running)
  • Engage business users early‚Äîthey depend on Workshop apps
‚úó Migration Don'ts
  • Don't assume 1:1 mapping‚ÄîOntology concepts don't directly translate
  • Don't underestimate Action complexity‚Äîvalidation logic is embedded everywhere
  • Don't migrate Workshop apps as-is‚Äîthey must be completely rebuilt
  • Don't forget about AIP workflows‚Äîthese need full reconstruction
  • Don't ignore lineage‚ÄîUnity Catalog won't automatically inherit Palantir lineage
  • Don't skip user training‚ÄîDatabricks UX is very different from Workshop
  • Don't migrate everything at once‚Äîphase by domain/use case
  • Don't lose audit history‚Äîexport Action logs before decommissioning

Key Risks & Considerations

Ontology Loss: The biggest risk is losing the semantic layer. Years of business logic encoded in Object Types, Link Types, and Actions must be re-implemented‚Äîoften in multiple places (dbt models, application code, API validators). There is no single Databricks equivalent.

Operational App Gap: Workshop applications that business users depend on daily have no direct replacement. Rebuilding in Tableau/Power BI covers dashboards but not operational workflows with forms, actions, and approvals. Custom development is often required.

AI Workflow Disruption: AIP Logic functions that automate Ontology operations must be rebuilt as Databricks AI Functions or external services. The tight integration between LLMs and business objects is lost.

Complexity Compounds: While individual ETL migrations may be straightforward, the cumulative complexity of rebuilding Ontology, Applications, and AI workflows simultaneously creates significant project risk. Plan phased approaches with clear milestones.
