Overview

Fundamentals of traffic distribution

What is Load Balancing?

Load balancing distributes incoming network traffic across multiple servers to ensure no single server bears too much load. This improves responsiveness, increases availability, and enables horizontal scaling.

Modern load balancers do more than traffic distributionâ€”they provide SSL termination, health monitoring, session persistence, and content-based routing.

Key Benefits

  • High Availability â€” Route around failed servers automatically
  • Scalability â€” Add servers without changing client configuration
  • Performance â€” Distribute load evenly, avoid hotspots
  • Security â€” Hide backend topology, terminate SSL at edge
  • Flexibility â€” Route based on URL, headers, or content

Types of Load Balancers

Hardware Load Balancers: Physical appliances (F5, Citrix) with dedicated ASICs. High performance but expensive and inflexible. Legacy in most cloud environments.

Software Load Balancers: Run on commodity hardware or VMs (NGINX, HAProxy, Envoy). Flexible, scriptable, and cost-effective. Dominant in modern architectures.

Cloud Load Balancers: Managed services (AWS ALB/NLB, Azure Load Balancer, GCP Load Balancer). Auto-scaling, integrated with cloud services, pay-per-use pricing.

When to Use Each

Use Layer 4 for: High-throughput TCP/UDP services, non-HTTP protocols (databases, MQTT, gaming), when you need absolute minimum latency, or passthrough SSL (end-to-end encryption without termination).

Use Layer 7 for: Web applications, APIs, microservices routing, when you need path-based routing (/api vs /static), A/B testing, canary deployments, or WAF integration.

Health Checks

Monitoring target availability

Why Health Checks Matter

Health checks verify that backend servers can handle requests. Without them, the load balancer would send traffic to crashed or overloaded servers, causing failures.

Configure checks to detect both server availability (is it running?) and application health (is it working correctly?).

Types of Health Checks

  • TCP â€” Connection succeeds (basic liveness)
  • HTTP â€” Returns expected status code (200)
  • HTTPS â€” HTTP over TLS
  • gRPC â€” gRPC health protocol
  • Custom â€” Application-specific endpoints

Use TLS 1.3 minimum (disable TLS 1.0/1.1). Enable HSTS headers. Use strong cipher suites (ECDHE, AES-GCM). Implement OCSP stapling for faster validation. Consider mutual TLS (mTLS) for service-to-service communication.

Best Practices

Guidelines for effective load balancing

1

Use Multiple Availability Zones

Deploy targets across at least 2-3 AZs for fault tolerance. Enable cross-zone load balancing to distribute traffic evenly. A single-AZ failure should not take down your application. Test failover scenarios regularly.

2

Implement Proper Health Checks

Use deep health checks that verify application functionality, database connectivity, and downstream dependenciesâ€”not just TCP port availability. Return 503 if the app can't serve traffic. Set intervals (30s) and thresholds (2-3 failures) appropriately.

3

Enable Connection Draining

Allow in-flight requests to complete before removing targets during deployments or scale-in. Set drain timeout (default 300s) based on your longest request duration. Prevents 5xx errors during rolling updates and auto-scaling events.

4

Configure Appropriate Timeouts

Set idle timeout based on application needs (default 60s). For long-polling or WebSockets, increase to 3600s. Configure backend response timeout shorter than client timeout. Use keep-alive for HTTP/1.1 connection reuse.

5

Use Sticky Sessions Sparingly

Prefer stateless backends with external session stores (Redis, DynamoDB). If stickiness is required, use application-controlled cookies over LB-generated ones. Duration-based stickiness can cause uneven load distribution and complicate scaling.

6

Monitor and Alert Proactively

Track key metrics: 5xx error rate (<0.1%), latency p99 (<500ms), healthy host count, request count, and connection count. Set up alerts for anomalies. Use access logs for debugging. Create dashboards for real-time visibility.

7

Pre-warm for Traffic Spikes

Cloud LBs scale automatically but need time. Contact AWS/GCP/Azure support 48hrs before expected traffic surges (product launches, marketing events, sales). Provide expected RPS, duration, and traffic pattern for optimal pre-warming.

8

Use TLS 1.3 and Strong Ciphers

Enable TLS 1.3 for faster handshakes and better security. Disable TLS 1.0/1.1 and weak ciphers. Use managed certificates (ACM, Let's Encrypt) with auto-renewal. Enable HSTS headers. Consider OCSP stapling for faster validation.
