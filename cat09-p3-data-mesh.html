Common Data Mesh Approaches

Implementation patterns and organizational models

Federated Full Decentralization
Each domain fully owns infrastructure, pipelines, and data products. Central team only provides standards and governance policies.
Best for: Mature orgs Strong engineering 100+ domains
Hybrid Platform + Domain Teams
Central platform provides compute, storage, catalog. Domain teams own data products and pipelines using shared infrastructure.
Best for: Mid-size orgs Mixed skill levels 20-100 domains
Embedded Platform Engineers in Domains
Platform engineers embedded in each domain team. They bring platform expertise while reporting to domain leadership.
Best for: Growing orgs Culture shift 5-50 domains
CoE Center of Excellence
Central CoE provides templates, training, and consulting. Domains own execution but get expert guidance on demand.
Best for: Transformation Skill building Any size

Data Science & AI with Data Mesh

Leveraging mesh architecture for ML and AI workloads

Pattern 1 ML as a Data Product
Treat trained models as data products with the same rigor: versioning, documentation, SLAs, and ownership. Models published through Unity Catalog with lineage tracking.
catalog.ml_domain.churn_model_v2 ‚Üí Returns predictions as a table consumers can query
Pattern 2 Feature Store as Shared Infrastructure
Centralized feature store enables feature reuse across domains. Data products feed features, ML domain consumes them. Prevents feature duplication and drift.
feature_store.customer.purchase_frequency_30d ‚Üí Reused by churn, recommendation, and fraud models
Pattern 3 Domain-Embedded ML Engineers
ML engineers sit within business domains, building models specific to domain needs. Central ML platform provides tools, domain provides context and data.
Sales Domain + ML Engineer ‚Üí Builds lead scoring model with deep sales context
Pattern 4 GenAI & RAG with Mesh
Vector stores as data products enable RAG patterns. Each domain publishes embeddings of their documents/data, consumed by GenAI applications.
catalog.legal_domain.contract_embeddings ‚Üí Powers contract Q&A chatbot

AI/ML Benefits in Data Mesh

  • Feature reuse: Features built once, consumed by many models
  • Data quality: Domain owners ensure training data accuracy
  • Faster iteration: ML teams don't wait for data team
  • Governance: Model lineage tracks back to source data products
  • Compliance: Know exactly what data trained each model

Key Technologies

  • Unity Catalog: Model registry + data governance unified
  • MLflow: Experiment tracking, model versioning, deployment
  • Feature Store: Online/offline serving, point-in-time correctness
  • Vector Search: Mosaic AI Vector Search for RAG
  • Model Serving: Real-time endpoints, batch scoring

The Four Principles of Data Mesh

Foundational concepts from Zhamak Dehghani

üè¢
Domain Ownership
Data owned by domain teams who understand the business context, not centralized data teams.
üì¶
Data as a Product
Treat data like a product with consumers, SLAs, discoverability, and quality guarantees.
üõ†Ô∏è
Self-Serve Platform
Infrastructure that enables domain teams to publish and consume data without central team dependency.
‚öñÔ∏è
Federated Governance
Global standards with local autonomy‚Äîcompliance enforced centrally, decisions made in domains.

Data Product Characteristics

  • Discoverable: Listed in catalog with clear description
  • Addressable: Unique identifier and access path
  • Trustworthy: Quality metrics and SLAs defined
  • Self-describing: Schema, lineage, ownership documented
  • Interoperable: Standard formats and protocols
  • Secure: Access controls and compliance built-in

Platform Capabilities

  • Data catalog: Discovery and metadata management
  • Compute provisioning: Self-serve Spark, SQL, ML
  • Storage management: Lakehouse with governance
  • Policy enforcement: Access, retention, quality
  • Observability: Lineage, quality monitoring
  • Data sharing: Cross-domain access controls

Best Practices

Guidelines for successful data mesh adoption

‚úì Do
  • ‚úì Start with high-value, well-understood domains first
  • ‚úì Define clear data product contracts with SLAs
  • ‚úì Invest heavily in self-serve platform capabilities
  • ‚úì Establish federated governance model early
  • ‚úì Embed data engineers in domain teams
  • ‚úì Use standard formats and protocols across domains
  • ‚úì Build data product thinking into culture
  • ‚úì Measure and publish data quality metrics
‚úó Don't
  • ‚úó Try to mesh everything at once‚Äîstart small
  • ‚úó Underinvest in the platform‚Äîit enables everything
  • ‚úó Expect domain teams to build from scratch
  • ‚úó Create data products without clear consumers
  • ‚úó Ignore organizational change management
  • ‚úó Skip governance‚Äîchaos ensues without standards
  • ‚úó Forget about cross-domain data products
  • ‚úó Treat data mesh as purely technical transformation
