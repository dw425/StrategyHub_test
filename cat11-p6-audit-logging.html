What is Audit Logging?

Audit logging is the practice of recording events that occur within systems, applications, and networks to create a chronological record of activities. These logs serve as the foundation for security monitoring, incident investigation, compliance reporting, and operational troubleshooting. Without comprehensive logging, organizations are blind to threats and unable to reconstruct events after incidents.

Why Logging Matters for Data Platforms

Data platforms are high-value targets because they contain sensitive information. Comprehensive logging provides visibility into who accessed what data, when, and how. This visibility is essential for detecting unauthorized access, investigating data breaches, meeting regulatory requirements (like GDPR's accountability principle), and maintaining trust with customers and stakeholders.

The Three Pillars of Security Logging

  • Detection: Identify security threats and anomalous behavior in real-time through continuous monitoring and alerting. Logs enable you to spot attacks as they happen.
  • Investigation: Reconstruct events during incident response to understand what happened, how it happened, and the extent of impact. Logs provide the forensic evidence needed for root cause analysis.
  • Compliance: Demonstrate adherence to security policies and regulatory requirements through audit trails. Logs prove to auditors that controls are operating effectively.

Logs vs. Metrics vs. Traces

Understanding the differences helps build comprehensive observability:

  • Logs: Discrete events with timestamps‚Äîsecurity events, errors, user actions. Answer "what happened?"
  • Metrics: Numerical measurements over time‚ÄîCPU usage, request rates, error counts. Answer "how much/many?"
  • Traces: End-to-end request paths across distributed systems. Answer "how did this flow through the system?"

What to Log: The Golden Rules

Not all events are equally important. Focus logging efforts on high-value security events:

  • Always log: Authentication events (all), authorization failures, privilege escalation, data access to sensitive tables, administrative changes, security tool alerts
  • Log selectively: Successful authorization (sample for high-volume), read access to non-sensitive data, routine automated operations
  • Never log: Passwords, API keys, tokens, credit card numbers, SSNs, or other secrets‚Äîeven in debug logs

Key Logging Metrics to Monitor

  • Log Collection Rate: Percentage of expected logs actually received. Drops indicate collection failures or agent issues.
  • Ingestion Latency: Time from event occurrence to searchability. Should be under 60 seconds for security use cases.
  • Alert-to-Triage Time: How quickly alerts are reviewed. Target under 15 minutes for critical alerts.
  • False Positive Rate: Percentage of alerts that aren't real threats. High rates cause alert fatigue‚Äîtarget under 10%.
  • Coverage Gaps: Systems or event types not being logged. Identify and remediate blind spots.

Architecture Best Practices

  • Centralize everything: All logs should flow to a single platform for correlation. Siloed logs create blind spots.
  • Normalize on ingest: Convert all logs to a common schema (like ECS or OCSF) to enable cross-source queries.
  • Buffer for reliability: Use message queues to prevent data loss during spikes or downstream outages.
  • Tier your storage: Hot storage for recent/frequent access, cold storage for long-term retention at lower cost.
  • Secure the pipeline: Logs often contain sensitive data‚Äîencrypt in transit and at rest, control access strictly.

Retention Strategy Considerations

  • Regulatory requirements: Different regulations mandate different retention periods. HIPAA requires 6 years, SOX 7 years, PCI-DSS 1 year. Always meet the strictest applicable requirement.
  • Litigation hold: When legal action is anticipated, suspend deletion of potentially relevant logs. Work with legal to define scope.
  • Cost optimization: Hot storage (fast SSD) costs 10-50x more than cold storage (S3 Glacier). Move logs to cheaper tiers as they age.
  • Investigation needs: Most investigations involve recent events, but APTs can have long dwell times. Balance cost vs. forensic capability.
  • Immutability: Archive logs should be write-once to prevent tampering. Use WORM storage or blockchain verification for legal admissibility.

Building Effective Detection Rules

  • Start with known attack patterns: Use MITRE ATT&CK framework as a guide for detection coverage.
  • Tune for your environment: Generic rules generate noise. Customize thresholds based on your baseline activity.
  • Layer detections: Single indicators are weak. Combine multiple signals for higher confidence alerts.
  • Reduce false positives: Every false positive erodes analyst trust. Continuously refine rules based on feedback.
  • Document and version: Treat detection rules as code. Use version control, testing, and change management.

Best Practices

Logging do's and don'ts

‚úì Do's
  • Log all authentication events‚Äîsuccessful and failed‚Äîwith source IP, user agent, and timestamp
  • Normalize logs to a common schema (ECS, OCSF) to enable cross-source correlation
  • Enrich logs with context: user department, asset criticality, data classification
  • Implement tamper-evident logging with cryptographic verification or WORM storage
  • Synchronize time across all systems using NTP to enable accurate correlation
  • Set up real-time alerting for high-priority security events with clear escalation
  • Conduct regular log reviews‚Äîdaily for critical systems, weekly for others
  • Test your logging pipeline regularly to ensure coverage and detect gaps
  • Document retention policies and automate lifecycle management
  • Protect log infrastructure as critical security assets‚Äîit's a prime attacker target
‚úó Don'ts
  • Log sensitive data: passwords, API keys, tokens, SSNs, credit card numbers‚Äîeven accidentally
  • Store logs only on the systems that generate them‚Äîcentralize to prevent tampering
  • Ignore log collection failures‚Äîtreat them as critical incidents requiring immediate action
  • Create alerts without tuning‚Äîalert fatigue leads to missed real threats
  • Delete logs before retention requirements expire or without legal review
  • Assume logging is working‚Äîverify coverage continuously through testing
  • Grant broad access to log data‚Äîit often contains sensitive operational details
  • Skip log analysis‚Äîcollected logs are worthless if never reviewed
  • Use inconsistent timestamps‚Äîtime zones and formats must be standardized
  • Treat logging as an afterthought‚Äîdesign it into systems from the start
