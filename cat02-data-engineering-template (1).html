Data Architecture Patterns

Compare data lake, warehouse, and lakehouse architectures

üåä
Data Lake
Schema-on-read storage
Raw data storage at scale. Store all data types in native format, apply schema when querying. Ideal for data exploration and ML workloads.
  • SchemaOn-read (flexible)
  • Data TypesAll (structured, semi, unstructured)
  • Costüí∞ Low storage cost
  • Query Speed‚ö° Variable
  • Best ForML, exploration, raw storage
  • ExamplesS3, ADLS, GCS
üè≠
Data Warehouse
Schema-on-write analytics
Optimized for analytical queries. Pre-defined schemas, ACID transactions, fast SQL performance. Powers BI dashboards and reports.
  • SchemaOn-write (strict)
  • Data TypesStructured only
  • Costüí∞üí∞üí∞ Higher (compute)
  • Query Speed‚ö°‚ö°‚ö° Fast SQL
  • Best ForBI, reporting, analytics
  • ExamplesSnowflake, BigQuery, Redshift
üè†
Data Lakehouse
Best of both worlds
Combines lake flexibility with warehouse performance. ACID transactions on object storage, supports both SQL analytics and ML workloads.
  • SchemaBoth (flexible + enforced)
  • Data TypesAll types supported
  • Costüí∞üí∞ Balanced
  • Query Speed‚ö°‚ö° Fast + flexible
  • Best ForUnified analytics + ML
  • ExamplesDatabricks, Delta Lake, Iceberg

Data Quality Dimensions

Measure, monitor, and improve data quality across the six core dimensions

‚úÖ Data Quality Framework ‚Äî Detection ‚Üí Measurement ‚Üí Remediation
üéØ Accuracy
Does the data correctly represent the real-world entity or event it describes?
HOW TO MEASURE:
  • ‚Ä¢ Sample audit vs source of truth
  • ‚Ä¢ Reconciliation reports (finance)
  • ‚Ä¢ User feedback / data corrections
COMMON ISSUES:
  • ‚Ä¢ Typos in manual entry
  • ‚Ä¢ Outdated reference data
  • ‚Ä¢ Incorrect data type conversions
‚úÖ Completeness
Is all required data present? Are there missing values, records, or attributes?
HOW TO MEASURE:
  • ‚Ä¢ Null/empty field percentage
  • ‚Ä¢ Row counts vs expected
  • ‚Ä¢ Required field coverage %
COMMON ISSUES:
  • ‚Ä¢ Optional fields left blank
  • ‚Ä¢ Dropped records in ETL
  • ‚Ä¢ Late-arriving data
üîÑ Consistency
Is the same data represented identically across different systems and tables?
HOW TO MEASURE:
  • ‚Ä¢ Cross-system reconciliation
  • ‚Ä¢ Duplicate key detection
  • ‚Ä¢ Hash comparison between sources
COMMON ISSUES:
  • ‚Ä¢ Different date formats
  • ‚Ä¢ Conflicting customer IDs
  • ‚Ä¢ Unsynchronized updates
‚è∞ Timeliness
Is data available when needed? Is it current enough for its intended use?
HOW TO MEASURE:
  • ‚Ä¢ Data freshness (age of records)
  • ‚Ä¢ SLA compliance %
  • ‚Ä¢ Pipeline latency tracking
COMMON ISSUES:
  • ‚Ä¢ Batch job delays
  • ‚Ä¢ Source system latency
  • ‚Ä¢ Time zone misalignment
‚úîÔ∏è Validity
Does data conform to defined formats, types, ranges, and business rules?
HOW TO MEASURE:
  • ‚Ä¢ Schema validation errors
  • ‚Ä¢ Business rule violations
  • ‚Ä¢ Regex pattern matching
COMMON ISSUES:
  • ‚Ä¢ Invalid email formats
  • ‚Ä¢ Out-of-range values
  • ‚Ä¢ Wrong enum/category values
üîë Uniqueness
Are there duplicate records that should be unique? Is the primary key truly unique?
HOW TO MEASURE:
  • ‚Ä¢ Duplicate key count
  • ‚Ä¢ Fuzzy matching for near-dupes
  • ‚Ä¢ Distinct ratio analysis
COMMON ISSUES:
  • ‚Ä¢ Double-loaded batches
  • ‚Ä¢ Merge/acquisition duplicates
  • ‚Ä¢ Weak identity resolution

    4. Configure monitoring and alerting
    5. Optimize for cost and performance
    """,
    agent=data_engineer,
    expected_output="Production-ready pipeline with 99.9% SLA"
)
