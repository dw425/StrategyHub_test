Overview

Understanding LLM observability and evaluation

What is LLM Observability?

LLM Observability is the practice of collecting, analyzing, and acting on data about your AI system's behavior in production. Unlike traditional software where you monitor uptime and error rates, LLM systems require monitoring output quality, hallucination rates, latency distributions, token usage, and cost per query.

Observability answers: What did the model actually do? Why did it fail? How much did it cost? Is quality degrading over time?

What are Evals?

Evaluations (Evals) are systematic tests that measure AI system quality. Unlike unit tests with binary pass/fail, evals often measure quality on a spectrum‚Äîrelevance scores, factual accuracy, tone appropriateness. They run on test datasets before deployment and continuously in production.

Evals answer: Is this change an improvement? Will this prompt work in production? Is the model still performing as expected?

‚úì Why Observability Matters
  • Catch issues early: Detect quality regressions before users complain
  • Debug faster: Trace exactly what happened on any request
  • Control costs: Track token usage and optimize expensive calls
  • Build confidence: Know your system works before deploying changes
  • Comply with audits: Log all inputs/outputs for regulated industries
  • Improve continuously: Data-driven optimization of prompts and retrieval
‚ö† Common Challenges
  • Non-determinism: Same input can produce different outputs
  • Subjective quality: "Good" responses are hard to define automatically
  • Scale of data: Logging every token is expensive at high volume
  • Privacy concerns: User inputs may contain sensitive data
  • Eval maintenance: Test sets drift as product evolves
  • LLM-as-judge bias: Using LLMs to evaluate LLMs has limitations

What to Log
Essential data for debugging and analysis

Request Data

  • Trace ID: Unique identifier linking all spans
  • User ID: Who made the request (anonymized if needed)
  • Timestamp: When the request started
  • Input: User query or prompt (consider PII masking)
  • Context: Retrieved documents, conversation history
  • Model: Which model version was used

Response Data

  • Output: Model response text
  • Latency: Total and per-component timing
  • Tokens: Input/output token counts
  • Cost: Calculated cost for this request
  • Quality scores: Any automated eval metrics
  • Errors: Exceptions, rate limits, timeouts

Measure RAG retrieval quality: recall@k, precision@k, MRR. Requires labeled "relevant documents" for each test query.
Best for: RAG systems, search quality, context relevance

Building an Eval Suite
Creating effective test datasets

Dataset Construction

  • Sample production queries: Real user questions from logs
  • Create golden answers: Human-verified correct responses
  • Tag relevant docs: Which documents should be retrieved
  • Include edge cases: Ambiguous, adversarial, out-of-scope queries
  • Cover query types: Factual, how-to, comparison, multi-hop
  • Size: Start with 50-100, grow to 500+ for production

Running Evals

  • CI/CD integration: Run on every PR that touches prompts
  • Baseline comparison: Always compare against current production
  • Regression detection: Alert if scores drop below threshold
  • A/B analysis: Statistical significance for improvements
  • Version tracking: Link results to prompt/model versions
  • Failure review: Human review of failed test cases

Cost Optimization Strategies

  • Model tiering: Use cheaper models for simple queries, expensive for complex
  • Prompt compression: Reduce context length while preserving quality
  • Caching: Cache responses for repeated or similar queries
  • Batch processing: Combine requests where latency permits
  • Output limits: Set max_tokens appropriate to use case

Cost Alerting

  • Budget thresholds: Alert at 80% of monthly budget
  • Anomaly detection: Flag unusual spikes in token usage
  • Per-user limits: Prevent runaway usage from single users
  • Cost attribution: Track spend by feature, team, or customer
  • Forecast trending: Predict month-end spend from current rate

Best Practices

Effective observability and evaluation

1
Log Everything (Carefully)
Capture all inputs, outputs, and metadata. But implement PII masking, set retention policies, and consider sampling at high volume.
2
Eval Before You Ship
Run evals on every change that touches prompts, retrieval, or model versions. Block deploys that regress below baseline.
3
Monitor Production Continuously
Quality can drift even without code changes. Run daily evals on production samples. Alert on degradation.
4
Version Everything
Track prompt versions, model versions, eval dataset versions. You need to know exactly what produced a result.
5
Combine Eval Methods
Use heuristics for basic checks, LLM-as-judge for nuance, and human review for calibration and edge cases.
6
Close the Feedback Loop
Use production failures to expand eval datasets. Every bug becomes a test case. Continuously improve coverage.

Agent This

Automated observability and evaluation agent

üìä ObservabilityAgent

An intelligent agent that monitors your LLM systems, runs evaluations, identifies quality regressions, and generates improvement recommendations. Integrates with your observability platform to provide continuous automated oversight.

When to Deploy

  • Production monitoring: Continuous quality checks on live traffic
  • Pre-deploy validation: Automated eval gates in CI/CD
  • Incident response: Rapid diagnosis when issues arise
  • Cost optimization: Identify expensive patterns and suggest fixes
  • Dataset curation: Automatically expand eval sets from failures
TraceFetcher
Monitoring Tool
Retrieves traces from your observability platform. Filters by time, error status, latency thresholds, or custom attributes.
Output: Filtered trace data for analysis
EvalRunner
Quality Tool
Executes eval suites against model outputs. Supports LLM-as-judge, heuristic checks, and retrieval metrics.
Output: Pass/fail results with scores
FailureAnalyzer
Diagnosis Tool
Analyzes failed evals and production errors to identify root causes‚Äîbad retrieval, prompt issues, or model limitations.
Output: Failure categorization and fix suggestions
CostAnalyzer
Optimization Tool
Analyzes token usage patterns and identifies optimization opportunities‚Äîcaching candidates, prompt compression, model tiering.
Output: Cost reduction recommendations
DatasetExpander
Curation Tool
Converts production failures and edge cases into eval dataset entries. Generates golden answers for human review.
Output: New eval cases for review
AlertManager
Notification Tool
Sends alerts via Slack, PagerDuty, or email when quality metrics breach thresholds or anomalies are detected.
Output: Formatted alerts with context
observability_agent.py
from crewai import Agent, Task, Crew
from langchain_anthropic import ChatAnthropic
from obs_tools import (
    TraceFetcher, EvalRunner, FailureAnalyzer,
    CostAnalyzer, DatasetExpander, AlertManager
)

# Initialize the Observability Agent
obs_agent = Agent(
    role="LLM Observability Engineer",
    goal="""Monitor LLM systems, run evaluations,
    identify quality issues, and recommend fixes""",
    backstory="""You are an expert in LLM 
    observability, evaluation frameworks, and 
    production monitoring. You analyze traces,
    run evals, diagnose failures, and optimize
    cost and quality.""",
    tools=[
        TraceFetcher(),
        EvalRunner(),
        FailureAnalyzer(),
        CostAnalyzer(),
        DatasetExpander(),
        AlertManager()
    ],
    llm=ChatAnthropic(model="claude-sonnet-4-20250514"),
    verbose=True
)

# Daily monitoring task
daily_monitor = Task(
    description="""
    Run daily observability checks:
    
    1. Fetch traces from last 24 hours
    2. Run eval suite on sample of requests
    3. Identify any quality regressions
    4. Analyze cost trends and anomalies
    5. Generate report with recommendations
    
    Alert if:
    - Eval pass rate < 85%
    - P95 latency > 3 seconds
    - Error rate > 2%
    - Cost up > 20% week-over-week
    """,
    agent=obs_agent,
    expected_output="Daily observability report"
)
incident_response.py
# Incident response task
incident_task = Task(
    description="""
    Investigate quality incident:
    
    1. Fetch recent traces with errors
    2. Identify failure patterns
    3. Run targeted evals to confirm scope
    4. Diagnose root cause:
       - Retrieval issues?
       - Prompt problems?
       - Model behavior change?
    5. Recommend immediate fixes
    6. Create eval cases from failures
    """,
    agent=obs_agent,
    expected_output="Incident report with RCA"
)

# Pre-deploy validation
deploy_gate = Task(
    description="""
    Validate deployment candidate:
    
    1. Run full eval suite against candidate
    2. Compare scores to production baseline
    3. Flag any regressions > 5%
    4. Check latency impact
    5. Estimate cost change
    6. Generate deploy/no-deploy recommendation
    """,
    agent=obs_agent,
    expected_output="Deploy gate decision"
)

async def run_monitoring():
    crew = Crew(
        agents=[obs_agent],
        tasks=[daily_monitor],
        verbose=True
    )
    return await crew.kickoff_async()
