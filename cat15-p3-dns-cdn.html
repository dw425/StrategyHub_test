Overview

Understanding DNS and content delivery networks

What is DNS?

The Domain Name System (DNS) translates human-readable domain names (example.com) into IP addresses (93.184.216.34) that computers use to route traffic. It's the internet's phonebookâ€”critical infrastructure that every request depends on.

Modern DNS does more than simple lookups: geographic routing sends users to nearby servers, health checks route around failures, and weighted policies enable gradual deployments and A/B testing.

What is a CDN?

A Content Delivery Network (CDN) caches and serves content from edge locations close to users. Instead of every request traveling to your origin server, static assets (images, CSS, JS) are served from the nearest edge point of presence (POP).

Benefits include: lower latency (content served closer to users), reduced origin load (70%+ requests served from cache), DDoS protection (attacks absorbed at edge), and global scale without infrastructure in every region.

How DNS and CDN Work Together

DNS and CDN are complementary technologies that work together for optimal content delivery. When a user requests your site, DNS resolves your domain to the nearest CDN edge location using anycast or geographic routing. The edge POP then serves cached content directly or fetches from origin if needed. This combinationâ€”intelligent DNS routing plus edge cachingâ€”is the foundation of modern global content delivery. Major CDN providers (CloudFront, Cloudflare, Akamai, Fastly) integrate DNS and CDN into unified platforms for seamless global distribution.

Recursive vs Iterative

Recursive: Resolver does all the work, querying each level and returning the final answer. Used by end clients. The resolver caches results for the TTL duration.

Iterative: Each server returns a referral to the next level. Used between DNS servers. More control but requires multiple round trips.

TTL (Time to Live)

TTL determines how long DNS responses are cached. Lower TTL (60s) enables faster changes but increases query load. Higher TTL (86400s) reduces queries but delays propagation.

Best practice: Use moderate TTL (300-3600s) normally, lower to 60s before planned changes, then raise again after propagation.

Cache-Control Directives

  • public â€” Can be cached by CDN and browser
  • private â€” Browser only, not CDN
  • max-age â€” Seconds content is fresh (browser)
  • s-maxage â€” Seconds for shared caches (CDN)
  • no-cache â€” Must revalidate before serving
  • no-store â€” Never cache anywhere
  • immutable â€” Content will never change

Best Practices

Guidelines for DNS and CDN optimization

1

Use Appropriate TTLs

Match TTL to content change frequency. Static assets: 1 year with cache-busting URLs. Dynamic content: 5-15 minutes. Lower TTL before planned changes, raise after. Default 300s is a good starting point for most records.

2

Implement Health Checks

Configure DNS health checks to route around failures automatically. Check application endpoints (/health), not just TCP. Set appropriate intervals (30s) and thresholds (2-3 failures). Test failover regularly.

3

Maximize Cache Hit Ratio

Target 90%+ cache hit ratio for static content. Use consistent URLs, remove unnecessary query strings from cache key. Analyze cache misses to identify optimization opportunities. Enable Origin Shield for popular content.

4

Use Cache-Busting for Deploys

Include content hash in filenames (app.a1b2c3.js) rather than query strings. Enables long TTLs while ensuring users get new versions. Works with all CDNs and proxies. Most build tools support this automatically.

5

Enable Compression

Enable Brotli (preferred) and Gzip at the CDN edge. Reduces bandwidth 60-80% for text content. Most CDNs handle this automatically based on Accept-Encoding header. Don't compress already-compressed formats (JPEG, PNG).

6

Implement DNSSEC

Enable DNSSEC to cryptographically sign DNS records, preventing spoofing and cache poisoning attacks. Most managed DNS providers support one-click enablement. Also configure CAA records to restrict certificate issuance.

7

Use Multiple DNS Providers

For critical applications, use multiple DNS providers (e.g., Route 53 + Cloudflare) for redundancy. Requires careful NS record management and synchronization. Consider DNS failover services for automatic switching.

8

Monitor and Alert

Set up monitoring for DNS resolution time, cache hit ratio, origin errors, and bandwidth. Alert on anomalies like sudden cache hit drops or increased origin load. Use real user monitoring (RUM) for actual user experience data.
