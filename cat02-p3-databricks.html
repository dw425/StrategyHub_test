What is Databricks?

The lakehouse company

The Lakehouse Platform

Databricks pioneered the "lakehouse" architecture‚Äîcombining the best of data lakes (cheap storage, schema flexibility, diverse data types) with the best of data warehouses (ACID transactions, governance, performance). Built on Apache Spark, Databricks provides a unified platform for data engineering, data science, machine learning, and analytics.

Founded by Spark Creators

Databricks was founded in 2013 by the original creators of Apache Spark at UC Berkeley. They went on to create Delta Lake (reliable data lake storage), MLflow (ML lifecycle), and Koalas (pandas on Spark). Today, Databricks is one of the most valuable private companies in tech, serving over 10,000 enterprise customers.

Multi-Cloud Native

Databricks runs on AWS, Azure, and GCP, with deep integrations into each cloud ecosystem. Compute runs in your cloud account, data stays in your storage (S3, ADLS, GCS), and Databricks provides the management plane. This separation enables cost efficiency and data residency compliance.

Delta Lake

Reliability and performance for data lakes

üîí
ACID Transactions
Serializable isolation level
Full ACID compliance on cloud object storage. Concurrent reads and writes without corruption. No more partial failures or inconsistent data.
  • Atomic commits across multiple files
  • Concurrent write safety with OCC
  • Consistent reads during writes
‚è±Ô∏è
Time Travel
Query historical data
Access and restore previous versions of your data. Query data as of any timestamp or version. Audit changes and rollback mistakes.
  • SELECT * FROM table VERSION AS OF 5
  • SELECT * FROM table TIMESTAMP AS OF '2024-01-01'
  • RESTORE TABLE table TO VERSION AS OF 3
üìä
Schema Evolution
Adapt to changing data
Add new columns, merge schemas automatically, and handle schema drift without breaking pipelines. No more failed jobs from schema changes.
  • Auto-merge compatible schemas
  • Add columns without rewriting data
  • Explicit schema enforcement option
‚ö°
Performance
Fast queries at scale
Z-ordering for data skipping, file compaction, caching, and Photon engine for 2-8x faster queries than open-source Spark.
  • Z-ORDER BY for multi-dimensional clustering
  • OPTIMIZE for file compaction
  • Photon vectorized execution engine
Delta Lake Operations
SQL / PySpark
-- Create a Delta table
CREATE TABLE customers (
    customer_id BIGINT,
    name STRING,
    email STRING,
    created_at TIMESTAMP
) USING DELTA
PARTITIONED BY (created_at)
LOCATION 's3://bucket/customers';

-- MERGE (upsert) pattern
MERGE INTO customers AS target
USING updates AS source
ON target.customer_id = source.customer_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- Time travel query
SELECT * FROM customers VERSION AS OF 5;

-- Optimize and Z-order for performance
OPTIMIZE customers ZORDER BY (customer_id);

Unity Catalog

Unified governance for the lakehouse

3-Level Namespace
Hierarchical organization
Organize data in a clear hierarchy: catalog.schema.table. Share data across workspaces while maintaining governance.
  • Catalog (top level, often per env)
  • Schema (database equivalent)
  • Tables, Views, Functions, Models
Fine-Grained Access
Row and column level security
Control who can see what data with row filters and column masks. Apply policies that follow data across queries and workspaces.
  • Row-level security filters
  • Column masking for PII
  • Dynamic views for access control
Data Lineage
Track data flow automatically
Automatic lineage tracking at table and column level. See where data comes from and where it goes‚Äîno manual documentation needed.
  • Automatic column-level lineage
  • Cross-workspace tracking
  • Impact analysis for changes

Delta Live Tables (DLT) Pipeline
Python / SQL
import dlt
from pyspark.sql.functions import *

# Bronze: Raw ingestion
@dlt.table(
    comment="Raw orders from source"
)
def bronze_orders():
    return spark.readStream.format("cloudFiles") \
        .option("cloudFiles.format", "json") \
        .load("/mnt/raw/orders")

# Silver: Cleaned and validated
@dlt.table(
    comment="Cleaned orders"
)
@dlt.expect_or_drop("valid_amount", "amount > 0")
@dlt.expect_or_fail("valid_id", "order_id IS NOT NULL")
def silver_orders():
    return dlt.read_stream("bronze_orders") \
        .select("order_id", "customer_id", "amount", "created_at")

# Gold: Business aggregates
@dlt.table(
    comment="Daily order metrics"
)
def gold_daily_metrics():
    return dlt.read("silver_orders") \
        .groupBy(to_date("created_at").alias("date")) \
        .agg(count("*").alias("order_count"), 
             sum("amount").alias("total_revenue"))

Best Practices

Guidelines for Databricks success

‚úì Do's
  • Use Unity Catalog for all new projects‚Äîit's the future of governance
  • Leverage Delta Lake everywhere‚Äîtables, streaming, ML feature stores
  • Use job clusters for production‚Äîephemeral, isolated, cost-effective
  • Implement medallion architecture (bronze/silver/gold)
  • Enable Photon for SQL-heavy workloads (2-8x speedup)
  • Use OPTIMIZE and Z-ORDER for frequently queried columns
  • Implement DLT for streaming pipelines with quality expectations
  • Set up proper cluster policies to control costs
‚úó Don'ts
  • Don't use all-purpose clusters for production jobs
  • Avoid small files‚Äîuse Auto Loader and OPTIMIZE regularly
  • Never skip VACUUM‚Äîold files accumulate and cost money
  • Don't over-partition‚Äîtoo many partitions hurt performance
  • Avoid legacy Hive metastore for new projects
  • Don't ignore cluster auto-termination settings
  • Never hardcode secrets‚Äîuse Databricks secrets or external vaults
  • Don't skip data quality checks‚Äîuse DLT expectations or Great Expectations
