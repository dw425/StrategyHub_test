Alerting Architecture

How alerts flow from detection to resolution

Modern alerting systems separate concerns: monitoring tools detect conditions, alert managers deduplicate and route, incident platforms manage response, and communication tools keep stakeholders informed. This separation enables flexibility and prevents vendor lock-in.

Alert Severity Levels

Standardized priority classification

Consistent severity levels ensure the right response to every incident. Critical issues page immediately, while lower severities may only notify during business hours. Define clear criteria for each level to prevent alert fatigue and ensure urgent issues get immediate attention.

P1 / SEV1
Critical
Complete service outage or data loss. Revenue-impacting. Requires immediate response 24/7.

Alert Anatomy

Components of a well-designed alert

üö® Alert Structure

alert_name
Clear, descriptive name. Example: "HighErrorRate_PaymentService" not "Alert1"
severity
P1/Critical, P2/High, P3/Warning, P4/Info. Determines routing and response.
description
Human-readable explanation of what triggered and why it matters.
expression
The condition that fires the alert. PromQL, LogQL, or platform-specific query.
for_duration
How long condition must be true before firing. Prevents flapping. Example: 5m
labels
Metadata for routing: team, service, environment, tier. Used by alert manager.
runbook_url
Link to troubleshooting steps. Critical for fast resolution by on-call.
dashboard_url
Direct link to relevant dashboard for investigation.

On-Call Practices

Sustainable and effective on-call rotations

üìÖ
Rotation Design
  • 1-week rotations balance context and burnout
  • Minimum 3-4 people per rotation for coverage
  • Primary + secondary on-call for escalation
  • Handoff meetings at rotation start
  • Time off after heavy incident weeks
  • Compensate for after-hours pages
üì±
Response Expectations
  • Acknowledge P1/P2 within 5 minutes
  • Have laptop and internet access available
  • Know how to reach secondary and leads
  • Follow runbooks before escalating
  • Document actions in incident channel
  • Escalate early if stuck or overwhelmed
‚¨ÜÔ∏è
Escalation Policies
  • Auto-escalate if no ack in 10 minutes
  • Escalate to secondary, then team lead
  • P1: Escalate to incident commander immediately
  • Cross-team: Know who owns dependencies
  • Executive escalation path for major outages
  • Never let pages go unanswered
üßò
Sustainable On-Call
  • Target <2 pages per on-call week
  • Fix noisy alerts immediately
  • Track on-call load per person
  • Rotate through all team members fairly
  • Budget time for reliability improvements
  • Celebrate good on-call weeks

Runbooks

Actionable documentation for incident response

Every alert should link to a runbook. Runbooks provide step-by-step troubleshooting guides that enable any on-call engineer to diagnose and resolve issues quickly, even if they're unfamiliar with the specific service.

üìã Runbook Template

# Alert: HighErrorRate_PaymentService

## Summary
Payment service error rate exceeds 1% for 5+ minutes.
This affects checkout flow and revenue.

## Severity
P1 - Revenue impacting. Page immediately 24/7.

## Dashboard
https://grafana.internal/d/payment-service

## Quick Diagnosis
1. Check error rate dashboard - is it rising or stable?
2. Check recent deployments: `kubectl rollout history`
3. Check dependent services: Database, Redis, Stripe API
4. Check for traffic anomalies (DDoS, bot traffic)

## Common Causes & Fixes

### Bad Deployment
- Symptoms: Errors started exactly at deploy time
- Fix: `kubectl rollout undo deployment/payment-service`

### Database Connection Pool Exhausted
- Symptoms: "connection timeout" in logs
- Fix: Restart pods: `kubectl rollout restart deployment/payment-service`
- Long-term: Increase pool size in config

### Stripe API Degradation
- Symptoms: Stripe errors in logs, status.stripe.com shows issues
- Fix: Enable payment queue, notify customers

## Escalation
- If not resolved in 15 min: Page @payments-lead
- If data integrity concerns: Page @database-team
- If suspected security issue: Page @security-oncall

## Post-Incident
- Create incident ticket in Jira
- Schedule postmortem if P1/P2
- Update this runbook if new failure mode found

SLO-Based Alerting

Alert on what matters to users

Instead of alerting on arbitrary thresholds, SLO-based alerting fires when you're burning through your error budget too fast. This approach focuses on user impact and prevents alert fatigue from non-impacting issues.

99.9%
Availability SLO
43.8 minutes of downtime allowed per month. Alert when burn rate exceeds 14.4x (1h window).
200ms
Latency P99 SLO
99% of requests must complete in 200ms. Alert when burn rate exceeds budget consumption.
0.1%
Error Budget
Monthly budget for failures. When exhausted, freeze deployments and focus on reliability.

üî• Burn Rate Alert Example

Fast Burn (P1)
14.4x burn rate for 1 hour = 2% budget consumed. Exhausts monthly budget in 2.5 days. Page immediately.
Slow Burn (P2)
6x burn rate for 6 hours = 5% budget consumed. Exhausts monthly budget in 5 days. Ticket, investigate.

Best Practices

Guidelines for effective alerting

1

Every Alert Must Be Actionable

If there's nothing to do, it shouldn't page. Convert to dashboard or remove entirely.

2

Every Alert Needs a Runbook

Link to troubleshooting steps. Enable any on-call engineer to resolve, not just experts.

3

Alert on Symptoms, Not Causes

Users care about errors and latency, not CPU. High CPU alone doesn't mean broken.

4

Use Appropriate Thresholds

Add duration (for: 5m) to prevent flapping. Set thresholds based on real impact, not arbitrary numbers.

5

Implement Alert Deduplication

Group related alerts. Don't page 50 times for one incident. Use alert manager grouping.

6

Review Alerts Regularly

Quarterly alert review. Delete unused alerts. Fix noisy ones. Track alert volume trends.

7

Blameless Postmortems

Focus on systems, not individuals. Ask "how did the system allow this?" not "who caused this?"

8

Track Incident Metrics

MTTD, MTTA, MTTR. Track trends over time. Set targets and celebrate improvements.

Anti-Patterns to Avoid

Common alerting mistakes

üîî Alert Fatigue
Too many alerts, most non-actionable. Team ignores pages. Real issues get missed.
Every alert actionable. Delete or convert to dashboard if not. Target <2 pages/week.
üñ•Ô∏è Alerting on Resource Metrics
Paging for high CPU or memory when service is healthy. Wastes time investigating non-issues.
Alert on symptoms (errors, latency). Use resource metrics for capacity planning only.
üìã No Runbooks
Alerts fire with no guidance. On-call spends time figuring out what to do instead of fixing.
Every alert links to runbook. Update runbooks after each incident with new learnings.
üë§ Hero Culture
Same person always handles incidents. Knowledge silos. Burnout. Team doesn't learn.
Rotate on-call fairly. Document everything. Pair on incidents for knowledge sharing.
üîá Permanent Silences
Silencing alerts indefinitely instead of fixing. Known issues accumulate. Eventually bite back.
Time-limit all silences. Create tickets for silenced alerts. Review and resolve.
üëÜ Blame-Based Postmortems
Focusing on who made the mistake. People hide issues. Learning doesn't happen.
Blameless culture. Focus on system improvements. "How do we prevent this class of error?"
