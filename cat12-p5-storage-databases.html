Data Persistence Fundamentals

Understanding cloud storage and database options

Storage vs. Database

Storage is raw persistence‚Äîfiles, blobs, blocks. No query engine, no schema, no relationships. You decide how to organize and access data.

Databases add structure and query capabilities. Schemas, indexes, transactions, and query languages help you work with data efficiently.

Most applications use both: databases for structured application data, storage for files, backups, and large objects.

Key Selection Criteria

Access Patterns: Random reads? Sequential scans? Write-heavy? Read-heavy? Access pattern determines optimal storage/DB type.

Consistency Requirements: Strong consistency (banking) vs. eventual consistency (social media). Drives database architecture choices.

Scale Requirements: Terabytes vs. petabytes. Thousands vs. millions of IOPS. Different solutions for different scales.

Cost Sensitivity: Storage tiers and DB instance sizing dramatically impact costs. Right-size from day one.

Managed vs. Self-Managed

Cloud providers offer managed services for most storage and database needs. Managed services handle patching, backups, replication, and scaling‚Äîyou focus on your application. Self-managed (databases on VMs) gives maximum control but requires operational expertise. Default to managed unless you have specific requirements (custom extensions, specific versions, licensing) that mandate self-management. The operational overhead of self-managed databases is often underestimated‚Äîfactor in 24/7 on-call, patching windows, and upgrade complexity.

S3 Best Practices

Bucket Naming: Use DNS-compliant names. Include environment/purpose (prod-app-media).

Versioning: Enable for critical buckets. Protects against accidental deletion.

Lifecycle Policies: Auto-transition to cheaper tiers or delete old versions.

Encryption: SSE-S3 (default), SSE-KMS (audit), or SSE-C (client-managed).

Access: Block public access by default. Use VPC endpoints for private access.

EBS Best Practices

gp3 over gp2: gp3 is 20% cheaper and allows independent IOPS/throughput scaling.

Snapshots: Incremental and stored in S3. Frequent snapshots are cost-effective.

Encryption: Enable by default. No performance penalty, uses KMS.

Multi-Attach: io1/io2 only. Application must handle concurrent writes (cluster-aware).

Right-Size: Use CloudWatch metrics to identify underutilized volumes.

Azure Blob Best Practices

Storage Accounts: Use GPv2 for most workloads. Premium for low-latency requirements.

Access Tiers: Set default tier. Use lifecycle management for auto-tiering.

Immutability: WORM policies for compliance (legal hold, time-based retention).

Redundancy: LRS (3 copies local), ZRS (3 zones), GRS (cross-region), GZRS (best).

ADLS Gen2 for Analytics

Hierarchical Namespace: Enables true directory operations, critical for Spark/Databricks.

HDFS Compatible: Use abfs:// driver. Drop-in replacement for HDFS in many tools.

Fine-Grained ACLs: POSIX-style permissions at directory and file level.

Integration: Native with Synapse, Databricks, HDInsight, Data Factory.

Cloud Storage Best Practices

Bucket Location: Choose region (lower cost) or multi-region (higher availability).

Autoclass: Great for unknown patterns. Moves objects automatically, no lifecycle rules needed.

Object Versioning: Enable for critical data. Combine with lifecycle rules to manage versions.

Uniform Access: Use uniform bucket-level access (disable ACLs) for simpler IAM.

Persistent Disk Best Practices

pd-balanced Default: Best price/performance for most workloads including boot disks.

Regional PD: Synchronous replication across zones. Required for HA without manual snapshots.

Hyperdisk: Use Extreme for OLTP databases, Throughput for analytics. Scale independently.

Snapshots: Schedule via snapshot policies. Multi-regional for DR.

OCI Storage Advantages

Pricing: Often 30-50% cheaper than AWS/Azure for storage. Competitive with GCP.

Free Egress: No charge for data transfer within OCI or to internet (up to 10TB/mo).

Performance: Block volumes include baseline IOPS. Ultra High tier for demanding workloads.

Oracle DB Integration: Optimized for Oracle Database, Exadata, Autonomous DB.

HDFS vs. Cloud Object Storage

HDFS Strengths: Lowest latency for Hadoop jobs, data locality, well-understood.

Cloud Object Strengths: Infinite scale, durability, cost-effective, serverless compute integration.

Modern Approach: Use cloud object storage (S3/ADLS/GCS) as primary. HDFS for transient cluster scratch. Table formats (Iceberg/Delta) for ACID and governance.

Migration Path: HDFS ‚Üí Cloud Object + Table Format. Use S3A/ABFS connectors.

Lifecycle Policy Strategies

Logs: Standard ‚Üí IA (30d) ‚Üí Cold (90d) ‚Üí Archive (180d) ‚Üí Delete (365d).

Backups: Standard ‚Üí IA (7d) ‚Üí Cold (30d) ‚Üí Archive (90d) ‚Üí Keep 7 years.

User Uploads: Use intelligent tiering (auto-moves based on access patterns).

ML Training Data: Standard during training ‚Üí IA after model deployed.

Cost Optimization Tips

Right-Size Block: Most VMs are over-provisioned. Start smaller, scale up as needed.

Lifecycle Everything: Automate tier transitions. Manual management doesn't scale.

Delete Unused: Orphaned snapshots, old backups, unused volumes add up quickly.

Monitor Egress: Data transfer costs can exceed storage costs. Keep data close to compute.

CAP Theorem

Understanding distributed database trade-offs

Choose Two (During Network Partition)
C
Consistency
Every read receives the most recent write or an error
A
Availability
Every request receives a response (not error)
P
Partition Tolerance
System operates despite network failures

CP Systems (Consistency + Partition Tolerance)

Sacrifice availability for consistency. During partition, may reject writes to maintain consistency.

Examples: MongoDB (in strict mode), HBase, Redis Cluster, Zookeeper.

Use when: Banking, inventory, any system where stale reads are unacceptable.

AP Systems (Availability + Partition Tolerance)

Sacrifice consistency for availability. May serve stale data during partition, reconcile later.

Examples: Cassandra, CouchDB, DynamoDB (default), Riak.

Use when: Social media, shopping carts, any system where availability > perfect consistency.

In Practice: Tunable Consistency

Modern databases often let you tune consistency per operation. DynamoDB offers eventual or strong consistency per read. Cosmos DB has 5 consistency levels (Strong ‚Üí Eventual). Cassandra uses quorum settings. Choose the right consistency level for each use case‚Äîstrong for payments, eventual for timelines.

Database Migration

Strategies for moving databases to cloud

üîÑ
Homogeneous Migration
Same engine (Oracle ‚Üí Oracle, MySQL ‚Üí MySQL). Use native tools, minimal transformation needed.
  • Create target instance (RDS/Cloud SQL)
  • Set up replication (DMS continuous)
  • Validate data integrity
  • Cutover during maintenance window
  • Redirect application connections
üîÄ
Heterogeneous Migration
Different engines (Oracle ‚Üí PostgreSQL). Schema conversion, code changes, extensive testing required.
  • Assess schema compatibility (SCT)
  • Convert schemas and procedures
  • Modify application code (queries)
  • Migrate data with transformation
  • Extensive parallel testing
üìä
Dual-Write Migration
Write to both old and new databases during migration. Zero downtime but complex to implement.
  • Backfill historical data
  • Enable dual writes in app
  • Validate data consistency
  • Gradually shift reads to new DB
  • Disable writes to old DB
üì¶
Dump & Restore
Simple backup and restore. Works for small databases where downtime is acceptable.
  • Schedule maintenance window
  • Stop writes to source
  • Export full dump (pg_dump, mysqldump)
  • Import to target database
  • Update connection strings

RPO vs RTO

RPO (Recovery Point Objective): Maximum acceptable data loss. How much data can you afford to lose? Drives backup frequency.

RTO (Recovery Time Objective): Maximum acceptable downtime. How fast must you recover? Drives standby architecture.

Lower RPO/RTO = higher cost. Match to business criticality.

The 3-2-1 Rule

3 copies of your data (primary + 2 backups)

2 different storage types (local + remote)

1 copy offsite (different region or cloud)

For critical data, consider 3-2-1-1 (add 1 air-gapped or immutable copy).

Database-Specific Strategies

RDS/Cloud SQL: Automated backups enabled by default. Point-in-time recovery up to 35 days. Manual snapshots for long-term.

DynamoDB: PITR (Point-in-Time Recovery) for 35 days. On-demand backups for compliance. Global Tables for DR.

Self-Managed: Implement your own backup scripts. Use native tools (pg_dump, mongodump). Store in object storage.

Testing Your Backups

Regular Restore Tests: A backup you've never restored is not a backup. Test quarterly at minimum.

Full DR Drills: Annual failover tests to validate RTO. Document issues and improve.

Chaos Engineering: Randomly fail components to validate recovery. Use tools like Chaos Monkey.
