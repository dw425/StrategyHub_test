Overview

Understanding agentic AI systems

What is Agentic AI?

Agentic AI refers to AI systems that can autonomously plan, make decisions, use tools, and take actions to accomplish goalsâ€”going beyond simple question-answering to actually doing things in the world.

Unlike traditional chatbots that respond to queries, agents can: break down complex tasks into steps, decide which tools to use, execute actions, observe results, and iterate until the goal is achieved.

Key Characteristics

  • Goal-directed â€” Works toward objectives, not just responses
  • Tool use â€” Can call APIs, execute code, browse web
  • Planning â€” Breaks complex tasks into steps
  • Reasoning â€” Decides what to do based on context
  • Persistence â€” Maintains state across interactions
  • Self-correction â€” Detects and fixes errors

The Agent Paradigm Shift

We're moving from AI as oracle (ask questions, get answers) to AI as worker (assign tasks, get results). This is a fundamental shift in how we interact with AI systems. Instead of being tools we operate, agents become colleagues we delegate to. The implications for automation, productivity, and work itself are profoundâ€”agents can handle multi-step workflows that previously required human judgment and action at every step.

The Agent Loop

Most agents follow a fundamental loop: Perceive (observe context and results) â†’ Think (reason about what to do) â†’ Act (execute tool or action) â†’ Repeat until goal achieved or failure detected.

This loop runs until a termination condition: task complete, error threshold exceeded, human intervention requested, or resource limits hit.

Critical Design Decisions

  • Planning depth â€” Plan everything upfront vs plan-as-you-go
  • Error handling â€” Retry, escalate, or fail fast
  • State management â€” Stateless vs stateful agents
  • Human handoff â€” When and how to involve humans

MCP Architecture

MCP uses a client-server model: the AI application (client) connects to MCP servers that expose tools. Communication happens over JSON-RPC, with servers declaring their capabilities and clients invoking them.

This separation means you can build a tool once and use it everywhereâ€”across Claude, Cursor, custom apps, and any future MCP client.

Building MCP Servers

MCP servers are lightweight processes that expose tools via a standard interface. SDKs available for Python and TypeScript. A server defines:

  • Tools â€” Functions the agent can call
  • Resources â€” Data the agent can access
  • Prompts â€” Pre-built prompt templates

Tool Use & Function Calling

The foundation of agent capabilities

How Tool Use Works

The model receives a list of available tools with descriptions and schemas. When the model decides to use a tool, it outputs a structured function call with the tool name and arguments. Your code executes the tool and returns results to the model.

This creates a loop: model reasons â†’ calls tool â†’ observes result â†’ reasons again â†’ calls another tool (or responds).

Tool Design Principles

  • Clear descriptions â€” The model chooses tools based on descriptions
  • Constrained inputs â€” Use enums, validation, strong typing
  • Informative outputs â€” Return enough context for next decisions
  • Error messages â€” Help model understand and recover from failures
  • Idempotency â€” Safe to retry on failure

from crewai import Agent, Task, Crew, Process
from crewai_tools import (
    SerperDevTool, ScrapeWebsiteTool, FileReadTool,
    DirectoryReadTool, PDFSearchTool

from langchain_anthropic import ChatAnthropic

# Initialize Tools
search_tool = SerperDevTool()
scrape_tool = ScrapeWebsiteTool()
file_tool = FileReadTool()
pdf_tool = PDFSearchTool()

# Define Agents with Roles
researcher = Agent(
    role="Senior Research Analyst",
    goal="Find comprehensive, accurate information on any topic",
    backstory="""You are an expert researcher with 15 years 
    experience in market research and competitive intelligence. 
    You excel at finding primary sources and validating information 
    across multiple channels.""",
    tools=[search_tool, scrape_tool],
    llm=llm,
    verbose=True,
    allow_delegation=True
)

analyst = Agent(
    role="Data Analyst",
    goal="Synthesize research into actionable insights",
    backstory="""You are a quantitative analyst who excels at 
    finding patterns in data and presenting complex information 
    in clear, structured formats. You always cite sources.""",
    tools=[file_tool, pdf_tool],
    llm=llm,
    verbose=True
)

writer = Agent(
    role="Technical Writer",
    goal="Create compelling, well-structured reports",
    backstory="""You are an award-winning writer who transforms 
    complex research into readable, actionable documents. 
    You focus on clarity, structure, and practical recommendations.""",
    tools=[file_tool],
    llm=llm,
    verbose=True
)

# Define Tasks
research_task = Task(
    description="""Research the current state of {topic}.
    Find: market size, key players, recent developments, trends.
    Use multiple sources and validate key claims.""",
    expected_output="Detailed research notes with sources",
    agent=researcher
)

analysis_task = Task(
    description="""Analyze the research on {topic}.
    Identify: key insights, patterns, opportunities, risks.
    Create a structured analysis with supporting data.""",
    expected_output="Structured analysis document",
    agent=analyst,
    context=[research_task]  # Uses output from research
)

report_task = Task(
    description="""Write a comprehensive report on {topic}.
    Include: executive summary, key findings, recommendations.
    Format: professional, scannable, action-oriented.""",
    expected_output="Final research report (markdown)",
    agent=writer,
    context=[research_task, analysis_task],
    output_file="report.md"  # Auto-save to file
)

# Create and Run Crew

    agents=[researcher, analyst, writer],
    tasks=[research_task, analysis_task, report_task],
    process=Process.sequential,  # or Process.hierarchical
    verbose=True
)

# Execute
result = research_crew.kickoff(inputs={"topic": "AI agents in enterprise"})

LangGraph Deep Dive

State machine orchestration for complex agents

ğŸ”„

Why LangGraph?

LangGraph models agents as state machinesâ€”graphs where nodes are actions and edges are transitions. This makes complex flows explicit and debuggable. It's the framework of choice when you need precise control over agent behavior, checkpointing, human-in-the-loop, and production reliability.

Core Concepts

  • State â€” The data that flows through the graph
  • Nodes â€” Functions that transform state
  • Edges â€” Connections between nodes (can be conditional)
  • Graph â€” The complete workflow definition
  • Checkpointer â€” Persistence for state recovery

Key Advantages

  • Explicit control flow â€” See exactly how agent decisions branch
  • Persistence â€” Resume from any checkpoint
  • Human-in-loop â€” Pause for approval at any node
  • Streaming â€” Real-time updates as agent works
  • Debugging â€” Trace execution through the graph
  • Testing â€” Unit test individual nodes
langgraph_example.py - ReAct Agent with LangGraph
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode, tools_condition
from langgraph.checkpoint.memory import MemorySaver
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage
from typing import TypedDict, Annotated
import operator

# Define state schema
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]
    next_action: str

# Initialize model with tools
llm = ChatAnthropic(model="claude-sonnet-4-20250514")
tools = [search_tool, calculator_tool, file_tool]
llm_with_tools = llm.bind_tools(tools)

# Define node functions
def call_model(state: AgentState) -> AgentState:
    """Call the LLM to decide next action."""
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def should_continue(state: AgentState) -> str:
    """Decide if we should continue or end."""
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "tools"
    return "end"

# Build the graph
workflow = StateGraph(AgentState)

# Add nodes
workflow.add_node("agent", call_model)
workflow.add_node("tools", ToolNode(tools))

# Add edges
workflow.set_entry_point("agent")
workflow.add_conditional_edges(
    "agent",
    should_continue,
    {"tools": "tools", "end": END}
)
workflow.add_edge("tools", "agent")

# Compile with checkpointing
memory = MemorySaver()
app = workflow.compile(checkpointer=memory)

# Run the agent
config = {"configurable": {"thread_id": "user-123"}}
result = app.invoke(
    {"messages": [HumanMessage(content="Research AI agents")]},
    config=config
)

# Stream events
for event in app.stream({"messages": [...]}, config):
    print(event)

Building resilient agents

âš ï¸

Agents Fail in Creative Ways

Unlike traditional software with predictable failure modes, agents can fail in surprising ways: hallucinated tool calls, infinite loops, subtle logic errors, adversarial manipulation. Defensive programming is essential. Assume every step can fail and design for graceful degradation.

ğŸ”„
Retry with Backoff

Transient failures (API timeouts, rate limits) should trigger retries with exponential backoff. Set max retries to prevent infinite loops.

ğŸ”€
Fallback Tools

If primary tool fails, try alternatives. Web search fails? Try cached results. API down? Use local fallback. Always have a backup plan.

âœ‚ï¸
Circuit Breaker

After N consecutive failures, stop trying that tool/path. Prevent wasted tokens on broken integrations. Reset after cooldown period.

ğŸ‘¤
Human Escalation

When agent is stuck or uncertain, escalate to human. Don't let agents spin endlessly. Provide context for efficient human resolution.

ğŸ’¾
Checkpoint & Resume

Save state at each step. On failure, resume from last checkpoint rather than starting over. Essential for long-running tasks.

ğŸ“
Structured Error Responses

Security Considerations

Protecting against agent vulnerabilities

ğŸš¨

Agents Expand Attack Surface

Agents that can execute code, call APIs, and interact with systems create significant security risks. Prompt injection can manipulate agent behavior. Tool abuse can cause damage. Data leakage can expose sensitive information. Security must be built in from day one.

1

Principle of Least Privilege

Give agents only the permissions they need. Read-only database access if writes aren't required. Scoped API keys. No admin credentials.

2

Sandbox Code Execution

Run agent code in isolated containers (Docker, E2B, Modal). No access to host filesystem, network restrictions, resource limits.

3

Input/Output Filtering

Validate all inputs. Filter outputs for PII, credentials, sensitive data. Don't let agents leak information in responses.

4

Prompt Injection Defense

Separate system prompts from user input. Use delimiters. Instruct models to ignore override attempts. Monitor for manipulation patterns.

5

Action Allowlists

Define explicitly what actions agents CAN take, not what they can't. Deny by default. Whitelist approved operations.

6

Audit Logging

Log every tool call, decision, and action. Maintain immutable audit trail. Enable forensic analysis of agent behavior.

Production Considerations

Making agents reliable enough for real workloads

âš ï¸

Agents Are Harder Than Chatbots

Agents have more failure modes: tool errors, infinite loops, wrong tool selection, cascading mistakes, cost runaway, security vulnerabilities. Production agents require significantly more engineering rigor than simple LLM applications. Plan for this complexity.

1

Observability is Non-Negotiable

Log every tool call, every decision, every state transition. Use LangSmith, Langfuse, or similar. Without observability, debugging agent failures is impossible.

2

Guardrails at Every Level

Input validation, output filtering, tool call limits, cost caps, time limits. Defense in depthâ€”assume each layer will fail and have the next layer catch it.

3

Graceful Degradation

When agents fail, fail safely. Return partial results, escalate to humans, or fall back to simpler approaches. Never fail silently or catastrophically.

4

Sandboxing & Permissions

Run agents with minimal permissions. Sandbox code execution. Use separate credentials. Treat agents as untrustedâ€”because they can be manipulated.

5

Cost Controls

Set hard limits on token usage, tool calls, and execution time. Agents can enter expensive loops. Monitor spend in real-time with alerts.

6

Evaluation & Testing

Build evaluation datasets for your agent tasks. Test edge cases explicitly. Regression test when changing prompts or tools. Agents are notoriously sensitive to small changes.

7

Human Handoff

Design clear escalation paths. When should the agent ask for help? When should it stop? Make it easy for humans to take over mid-task.

8

Retry & Recovery

Tools fail. APIs timeout. Implement exponential backoff, circuit breakers, and checkpointing so agents can recover from transient failures.

Cost Optimization

Making agents economically viable

ğŸ’¸

Agents Are Expensive

A single agent run can consume 10-100x more tokens than a simple chat. Multi-step reasoning, tool results in context, retriesâ€”it adds up fast. At scale, an unoptimized agent can cost thousands of dollars per day. Cost optimization isn't optional; it's essential.

1

Right-Size Your Models

Use smaller models for simple tasks (routing, classification). Reserve expensive models (o1, Claude Opus) for complex reasoning. Tiered approach can cut costs 60%+.

2

Aggressive Context Pruning

Don't keep everything in context. Summarize tool results. Drop irrelevant conversation history. Keep only what's needed for the current step.

3

Caching Everywhere

Cache tool results, model responses, intermediate computations. Same search query twice? Return cached result. Prompt caching for repeated system prompts.

4

Early Termination

Detect when task is complete and stop. Don't let agents over-iterate. Set max iteration limits based on task complexity.

5

Batch Similar Tasks

Process similar requests together. Share context across batch. One search for 10 queries is cheaper than 10 separate searches.

6

Monitor & Alert

Track cost per agent run. Alert on anomalies. Identify expensive patterns. You can't optimize what you don't measure.

Best Practices

Guidelines for building effective agents

1

Start Simple, Add Complexity

Begin with a single-agent, few-tool setup. Add multi-agent, more tools, and autonomy incrementally. Complexity is the enemy of reliability.

2

Invest in Tool Quality

Agents are only as good as their tools. Well-designed tools with clear descriptions, constrained inputs, and informative outputs dramatically improve agent performance.

3

Make Reasoning Visible

Agents should "think out loud." This helps with debugging, builds user trust, and often improves decision quality. ReAct pattern is your friend.

4

Test with Real-World Chaos

Test tool failures, malformed inputs, adversarial prompts, edge cases. Agents fail in unexpected waysâ€”find them before users do.

5

Design for Human Oversight

Even autonomous agents need supervision. Build approval workflows, intervention points, and clear audit trails. Humans should always be able to take over.

6

Manage Context Carefully

Context window is precious. Summarize conversation history, prune irrelevant tool results, and keep the agent focused on the current task.

7

Use the Right Model for the Job

Not every agent task needs o1. Use smaller/cheaper models for simple routing, larger models for complex reasoning. Cost optimization matters at scale.

8

Version Everything

Prompts, tools, workflowsâ€”version control all of it. Agent behavior can change dramatically with small changes. You need to know what changed when things break.

9

Implement Structured Outputs

Use JSON schemas for all tool inputs and agent outputs. Structured outputs eliminate parsing errors and make agent behavior more predictable.

10

Build in Observability

Log every decision, tool call, and state transition. Use LangSmith, Langfuse, or custom logging. Without observability, debugging is impossible.

# Agent Architect - Meta-agent for building agents
from crewai import Agent, Task, Crew
from langchain_anthropic import ChatAnthropic
from agent_tools import (
    PatternSelector, ToolDesigner, PromptGenerator,
    CodeScaffolder, EvalCreator, GuardrailAdder
)

llm = ChatAnthropic(model="claude-sonnet-4-20250514")

# Initialize meta-agent tools
pattern_selector = PatternSelector(patterns=[
    "react", "plan-execute", "reflection", 
    "router", "multi-agent"
])
tool_designer = ToolDesigner()
prompt_gen = PromptGenerator()
scaffolder = CodeScaffolder(frameworks=["langgraph", "crewai"])
eval_creator = EvalCreator()
guardrail_adder = GuardrailAdder()

agent_architect = Agent(
    role="Agent System Architect",
    goal="""Design and implement optimal agentic 
    systems for complex automation tasks""",
    backstory="""Expert in agent architectures with deep 
    experience building production agentic systems. 
    Specializes in selecting appropriate patterns, 
    designing reliable tools, and creating robust 
    multi-agent orchestrations.""",
    llm=llm,
    tools=[
        pattern_selector, tool_designer, prompt_gen,
        scaffolder, eval_creator, guardrail_adder
    ],
    verbose=True
)

async def design_agent_system(requirements: str) -> dict:
    """Design complete agent system from requirements."""
    
    task = Task(
        description=f"""
        Design an agent system for: {requirements}
        
        1. ANALYSIS
           - Identify core tasks and subtasks
           - Determine autonomy level needed
           - List required integrations
        
        2. ARCHITECTURE
           - Select agent pattern (ReAct, Plan-Execute, etc)
           - Design tool schemas with descriptions
           - Define state management approach
        
        3. IMPLEMENTATION
           - Generate agent prompts
           - Scaffold code in LangGraph
           - Add guardrails and error handling
        
        4. EVALUATION
           - Create test cases
           - Define success metrics
           - Build eval dataset
        """,
        agent=agent_architect,
        expected_output="Complete agent system design"
    )
    
    crew = Crew(agents=[agent_architect], tasks=[task])
    result = await crew.kickoff_async()
    
    return {
        "architecture": result.architecture,
        "tools": result.tool_schemas,
        "prompts": result.prompts,
        "code": result.implementation,
        "tests": result.eval_dataset
    }
