Overview

Foundational concepts for cloud network design

What is Network Architecture?

Network architecture defines the structure, components, and connectivity patterns of your cloud infrastructure. It encompasses Virtual Private Clouds (VPCs), subnets, routing tables, gateways, and the relationships between them.

Good network architecture enables scalability (grow without redesign), security (defense in depth), performance (low latency paths), and cost efficiency (optimized data transfer).

Key Components

  • VPC/VNet â€” Isolated virtual network in the cloud
  • Subnets â€” Subdivisions of VPC IP address space
  • Route Tables â€” Rules directing network traffic
  • Gateways â€” Entry/exit points (Internet, NAT, VPN)
  • Security Groups/NACLs â€” Firewall rules
  • Peering/Transit â€” Inter-VPC connectivity

Why This Matters

Network architecture decisions made early are difficult and expensive to change later. IP address conflicts prevent VPC peering. Poor segmentation creates security vulnerabilities. Inadequate capacity planning leads to costly redesigns. Investing time in proper network design pays dividends throughout the infrastructure lifecycleâ€”enabling faster deployments, simpler troubleshooting, and more secure operations.

Subnet Strategies

Organizing network segments for security and scale

Public Subnets

Resources with direct internet access via Internet Gateway. Use for load balancers, bastion hosts, and NAT gateways.

Route: 0.0.0.0/0 â†’ IGW

Resources: ALB/NLB, NAT Gateway, Bastion hosts, Public-facing APIs

Private Subnets

Application workloads without direct internet access. Outbound internet via NAT Gateway in public subnet.

Route: 0.0.0.0/0 â†’ NAT-GW

Resources: EC2, EKS nodes, Lambda, containers, backend services

Data Subnets

Isolated tier for databases and caches. No internet accessâ€”only internal communication.

Route: Local only (VPC CIDR)

Resources: RDS, ElastiCache, DocumentDB, Elasticsearch

Sizing Guidelines

  • /24 subnets â€” 251 usable IPs, good for most workloads
  • /23 subnets â€” 507 usable IPs, large EKS clusters
  • /22 subnets â€” 1,019 usable IPs, massive scale
  • AWS reserves 5 IPs per subnet (first 4 + last 1)
  • Leave room for growthâ€”subnets can't be resized

RFC 1918 Private Ranges

  • 10.0.0.0/8 â€” 16M addresses (Class A)
  • 172.16.0.0/12 â€” 1M addresses (Class B)
  • 192.168.0.0/16 â€” 65K addresses (Class C)

Use 10.x.x.x for cloud VPCs. Reserve 172.16.x.x for on-premises to avoid conflicts during hybrid connectivity.

Planning Tips

  • Document all CIDR allocations in a central registry
  • Plan for 3x current capacity to accommodate growth
  • Reserve IP blocks for future regions/environments
  • Avoid overlapping with partner/acquired networks
  • Use IPAM tools for large-scale management

Best Practices

Guidelines for robust network architecture

1

Plan CIDR Ranges Carefully

Document allocations, avoid overlaps with on-prem and partners. Leave room for 3x growth. Use /16 for production VPCs.

2

Use Multiple Availability Zones

Deploy subnets across at least 2 AZs (3 recommended). Single-AZ deployments are single points of failure.

3

Implement 3-Tier Segmentation

Separate public, private, and data tiers. Each tier has different routing and security requirements.

4

Use VPC Flow Logs

Enable flow logs for visibility into traffic patterns, troubleshooting, and security analysis. Send to CloudWatch or S3.

5

Deploy NAT Gateway Per AZ

Avoid cross-AZ NAT traffic and single points of failure. Each AZ should have its own NAT Gateway.

6

Use Transit Gateway for Scale

Replace mesh peering with Transit Gateway when connecting more than 3-4 VPCs. Simpler management, transitive routing.

7

Tag Everything

Consistent tagging enables cost allocation, automation, and compliance. Include environment, owner, and project tags.

8

Use Infrastructure as Code

Define networks in Terraform or CloudFormation. Version control, peer review, and repeatable deployments.
