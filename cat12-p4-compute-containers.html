Compute Fundamentals

Understanding cloud compute options

The Compute Evolution

Cloud compute has evolved from simple VMs to a spectrum of options. Each layer trades control for convenience‚ÄîVMs give full OS control, containers abstract the OS, serverless abstracts infrastructure entirely.

The right choice depends on workload characteristics: state requirements, scaling patterns, team expertise, and cost optimization goals.

Most organizations use multiple compute models simultaneously‚ÄîVMs for legacy apps, containers for microservices, serverless for event processing. The key is matching each workload to its optimal compute model.

Key Decision Factors

Control vs. Convenience: VMs offer maximum control; serverless offers maximum convenience. Choose based on operational maturity and compliance requirements.

Scaling Model: Steady workloads favor VMs with reserved pricing. Spiky workloads favor auto-scaling containers or serverless.

State Management: Stateful apps need VMs or StatefulSets. Stateless apps can leverage any compute model.

Cold Start Sensitivity: Latency-critical apps may not tolerate serverless cold starts (100ms-3s). Containers or VMs provide consistent response times.

Total Cost of Ownership Considerations

Compute costs extend beyond hourly rates. Factor in: Operations overhead (K8s requires expertise, serverless is hands-off), Licensing (Windows VMs cost more, BYOL can save 40%+), Data transfer (egress charges add up fast), Reserved capacity (commit for 30-72% savings), and Right-sizing (most VMs are over-provisioned by 40%+). A well-optimized container deployment often costs less than poorly-managed VMs, even if the per-unit container price appears higher.

Choosing Your Abstraction Level

Start with constraints: Compliance requirements, existing skills, application architecture, and vendor lock-in tolerance will often eliminate options before you evaluate technical fit.

Consider the lifecycle: VMs are great for lift-and-shift. Containers suit greenfield microservices. Serverless excels for new event-driven workloads.

Hybrid Approaches

Containers on VMs: Run Kubernetes on EC2/VMs for maximum control over the infrastructure while gaining container benefits.

Serverless containers: Services like Fargate, Cloud Run, and Container Apps give container portability with serverless operations.

VM Pricing Strategies

On-Demand: Maximum flexibility, pay by second/hour. Best for unpredictable, short-term, or spiky workloads.

Reserved/Committed: 1-3 year commitment for 30-72% savings. Best for steady-state production workloads.

Spot/Preemptible: Up to 90% discount using spare capacity. Can be interrupted. Best for fault-tolerant batch processing.

Savings Plans (AWS): Flexible commitment across instance families. Easier than reserved instances.

Right-Sizing Best Practices

Monitor Before Committing: Run on-demand for 2-4 weeks to understand actual usage patterns before reserving.

Use Provider Tools: AWS Compute Optimizer, Azure Advisor, GCP Recommender identify oversized instances.

Start Small: It's easier to scale up than to scale down. Begin with smaller instances and upgrade as needed.

Consider ARM: Graviton/Ampere instances offer 20-40% better price-performance for compatible workloads.

Container Image Best Practices

Use Minimal Base Images: Alpine (5MB), Distroless, or scratch. Smaller images = faster pulls + smaller attack surface.

Multi-Stage Builds: Compile in one stage, copy only artifacts to final image. Don't ship build tools.

Pin Versions: Never use :latest in production. Pin specific versions (node:18.17.0-alpine, not node:latest).

Layer Caching: Order Dockerfile instructions from least to most frequently changing. COPY package.json before COPY src/.

Container Registries

AWS ECR: Integrated with ECS/EKS, cross-region replication, image scanning included, lifecycle policies.

Azure ACR: Geo-replication, ACR Tasks for builds, integrated with AKS, content trust signing.

GCP Artifact Registry: Multi-format (Docker, npm, Maven), regional/multi-regional, vulnerability scanning.

Docker Hub: Public images, rate limits on free tier. Use for base images, not production private images.

Kubernetes Best Practices

Resource Requests/Limits: Always set CPU and memory requests for predictable scheduling. Set limits to prevent runaway containers from affecting neighbors.

Health Checks: Configure liveness probes (restart unhealthy) and readiness probes (remove from service). Start with conservative timeouts.

Pod Disruption Budgets: Define minimum available replicas during voluntary disruptions like node upgrades.

Kubernetes Anti-Patterns

Privileged Containers: Avoid running as root or with privileged mode. Use security contexts to drop capabilities.

Latest Tags: Never use :latest in production. Pin specific versions for reproducibility and rollback capability.

Monolithic Clusters: Don't put everything in one cluster. Separate by environment, team, or blast radius requirements.

When to Use Serverless

Event Processing: S3 uploads, database changes, IoT events‚Äîreact to triggers without polling. Perfect for fan-out patterns.

APIs: REST/GraphQL backends that scale from zero to millions of requests automatically. Ideal for variable traffic.

Scheduled Tasks: Cron jobs, report generation, cleanup scripts without running servers 24/7. Pay only when executing.

Glue Code: Connect services, transform data, handle webhooks. Small functions that don't justify a full service.

When to Avoid Serverless

Long-Running: Workloads exceeding timeout limits (15-60 min) need containers or VMs.

Consistent High Load: 24/7 high-throughput may be cheaper on reserved VMs. Break-even is typically 20-30% utilization.

Complex State: Heavy stateful processing is better suited for containers with volumes.

Cold Start Sensitive: Real-time applications requiring consistent sub-100ms latency should use containers or provisioned concurrency.

Compute Selection Guide

Match workloads to compute models

üåê

Traditional web apps with consistent traffic. Need session state, caching, and predictable performance.
Recommended: Containers on Kubernetes or managed container service (ECS, Cloud Run)

ETL processing, batch jobs, data transformation. Spiky workloads with periods of inactivity.
Recommended: Serverless (Lambda/Functions) with Step Functions for orchestration
ü§ñ

GPU-intensive model training. Long-running jobs requiring high memory and specialized hardware.
Recommended: GPU VMs (p4, NC-series) or managed ML platforms (SageMaker, Vertex AI)
üîå

REST/GraphQL APIs with variable traffic. Need auto-scaling and pay-per-request pricing.
Recommended: Serverless (Lambda + API Gateway) or Cloud Run

Recommended: VMs with attached storage or managed database services (RDS, Cloud SQL)

Container Security & CI/CD

Secure the container lifecycle from build to runtime

üõ°Ô∏è
Image Security
Build-time protection
Secure container images before they reach production. Shift security left into CI/CD pipelines.
  • Use minimal base images (Alpine, Distroless)
  • Scan for CVEs (Trivy, Snyk, Prisma)
  • Sign images (Cosign, Notary)
  • No secrets in images‚Äîuse vaults
  • Multi-stage builds to reduce attack surface
üîê
Runtime Security
Production protection
Protect running containers with runtime policies, network segmentation, and behavioral monitoring.
  • Pod Security Standards (restricted)
  • Network Policies for microsegmentation
  • Read-only root filesystems
  • Runtime threat detection (Falco)
  • Secrets from Vault/CSI driver
üîÑ
CI/CD Pipeline
Automated deployments
Automate build, test, scan, and deploy. GitOps for declarative, auditable deployments.
  • GitOps (ArgoCD, Flux)
  • Image promotion across environments
  • Canary/blue-green deployments
  • Automated rollback on failures
  • Policy-as-code (OPA Gatekeeper)

Supply Chain Security

SBOM (Software Bill of Materials): Generate and store SBOMs for all images. Required for compliance and incident response.

Trusted Registries: Only pull from approved registries. Block public Docker Hub in production.

Base Image Updates: Automate rebuilds when base images are patched. Dependabot for containers.

Compliance Frameworks

CIS Benchmarks: Follow CIS Docker and Kubernetes benchmarks. Automated scanning with kube-bench.

NIST 800-190: Application container security guide for federal systems and enterprises.

SOC 2 / PCI: Document container security controls, image scanning, and runtime protection in audit evidence.
