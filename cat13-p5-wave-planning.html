Wave Planning Overview

Strategic approach to phased migration execution

Wave planning is the strategic sequencing of migration workloads into manageable phases. Rather than attempting a "big bang" migration of all 4,231 objects simultaneously, we divide the work into 5 distinct waves based on complexity, dependencies, business criticality, and risk profile. This approach transforms an overwhelming undertaking into a series of achievable milestones, each building on the lessons and successes of the previous wave.

Each wave is a self-contained migration unit with its own assessment, conversion, validation, and cutover cycle. This approach allows for incremental value delivery, early identification of patterns and issues, and continuous improvement of migration processes across waves. Teams gain expertise and confidence with simpler objects before tackling the most complex and critical workloads in later waves.

Wave sequencing considers technical dependencies (tables referenced by views/procedures), business dependencies (related reporting domains), and organizational factors (team availability, change windows). The goal is to maximize learning in early waves while reserving complex, high-risk objects for later waves when processes are mature. We also balance each wave to include a mix of object types, ensuring that the team builds expertise across all conversion patterns.

Wave Details

Scope, timing, and composition of each wave

Each wave has a distinct scope and purpose, carefully designed to balance complexity, dependencies, and team capacity. The wave composition follows a "simple to complex" progression, with foundational objects in Wave 1 and the most challenging legacy code in Wave 5. Each wave includes a mix of tables, views, and stored procedures to ensure the team develops expertise across all object types.

Wave boundaries are drawn to minimize cross-wave dependencies while grouping related business domains together. For example, all customer-related dimension tables are in Wave 1, so the customer analytics in Wave 3 can rely on completed objects. The timing for each wave includes buffer for issue resolution, with approximately 1 week of overlap between waves to maintain momentum.

Wave Dependencies

Inter-wave relationships and sequencing constraints

Understanding wave dependencies is critical for schedule planning and risk management. The dependency matrix below shows which waves depend on others ("Depends" = prerequisite required) and which waves block subsequent waves ("Blocks" = must complete before successor can start). These dependencies are derived from object-level technical dependenciesâ€”if a view references a table, the table's wave must complete first.

Wave 1 has no dependencies and serves as the foundation for all subsequent waves. This was a deliberate design choice to minimize risk in the first wave. Wave 5 depends on all previous waves, which is why it's scheduled lastâ€”it contains objects that reference components from across the data platform. If Wave 2 experiences delays, Waves 3-5 are all impacted, making Wave 2's on-time completion the most critical path item.

Prioritization Criteria

Factors used to sequence objects into waves

Objects are assigned to waves using a weighted scoring model that balances multiple factors. Each object receives a composite score from 0-100 based on the criteria below, with lower scores assigned to earlier waves (simpler, lower-risk objects) and higher scores to later waves (complex, high-risk objects). The scoring is automated based on metadata from the assessment phase, with manual overrides for business-critical objects.

The weighting reflects our migration philosophy: technical complexity and dependencies are the primary drivers of wave assignment, ensuring we don't attempt to migrate objects before their prerequisites are ready. Business criticality is balanced against riskâ€”high-criticality objects go in middle waves (2-3) where processes are proven but not yet fatigued, rather than Wave 1 where tooling is still being validated.

Resource Allocation

Team assignments and capacity planning per wave

Resource allocation follows a ramp-up and taper pattern aligned with wave complexity. Wave 1 uses a smaller team as processes are established, ramping up to peak capacity in Wave 3 (highest technical complexity), then tapering in Waves 4-5 as patterns are established and automation handles more of the workload. QA resources scale with the volume of test cases, not just object count.

Compute resources (measured in DBUs) scale with data volume rather than object count. Wave 2 requires maximum Databricks capacity for transferring billion-row fact tables, while Wave 5 uses minimal compute despite having complex stored procedures. Storage allocation follows the same patternâ€”most of the 29 TB total is consumed in Waves 1-2 for dimension and fact tables, with later waves primarily adding views and procedures that reference existing data.

Risk Assessment

Wave-specific risks and mitigation strategies

Each wave has distinct risk characteristics based on object complexity, data volume, and business impact. Early waves carry higher learning curve riskâ€”the team is still refining processes and tooling. Later waves face deadline pressure and accumulated technical debt from issues deferred from earlier waves. Understanding these wave-specific risks allows targeted mitigation strategies.

The risk assessment is continuously updated as waves progress. Issues discovered in one wave inform risk mitigation for subsequent waves. For example, Wave 2 identified performance bottlenecks in the CDC sync process, leading to pre-emptive optimization for Wave 3 analytics tables. The overall program risk decreases as waves complete successfully and processes mature.

Agent This

AI-powered wave optimization

The WavePlannerAgent automates the complex task of optimal wave assignment by analyzing thousands of objects simultaneously, considering dependencies, complexity scores, and resource constraints. It can rebalance waves dynamically when scope changes or issues arise, ensuring the migration stays on track without manual replanning.

The agent integrates with the Assessment Crawler to receive object metadata, builds a dependency graph to identify critical paths, and uses constraint satisfaction algorithms to optimize wave assignments. It provides what-if analysis for schedule scenarios and alerts stakeholders when dependencies are at risk due to delays in upstream waves.
